# LeetCode — Construct Binary Tree from Inorder and Postorder Traversal

## Problem

Given two integer arrays:
- `inorder` — the inorder traversal (`left → root → right`) of a binary tree
- `postorder` — the postorder traversal (`left → right → root`) of the same tree

**Construct and return the binary tree.**

---

## Core Observations (How the Solution Was Constructed)

The solution is built on two fundamental properties of tree traversals:

### 1. Postorder's last element is always the current root
In postorder traversal, a node is visited *after* its children. So the **last element of the postorder array is the root** of the current (sub)tree. By repeatedly calling `postorder.pop()`, we discover each subtree's root in the correct order.

### 2. Inorder splits a tree into left and right subtrees
Once we know the root, its position in the inorder array divides the remaining nodes cleanly:
- Everything **to the left** → belongs to the **left subtree**
- Everything **to the right** → belongs to the **right subtree**

These two observations together make recursive construction possible.

---

## Construction Strategy

### Hashmap for O(1) Inorder Lookups
Rather than scanning the inorder array each time to locate the root, we precompute a value-to-index map:

```python
idx_map = {val: idx for idx, val in enumerate(inorder)}
```

This reduces each lookup from O(n) to O(1), keeping the overall complexity linear.

### Recursive Helper with Inorder Bounds
The helper function `helper(in_left, in_right)` operates on a logical window of the inorder array — no slicing, just index bounds.

```python
def helper(in_left, in_right):
    if in_left > in_right:
        return None             # No nodes left in this subtree

    val = postorder.pop()       # Current root (postorder: last = root)
    root = TreeNode(val)

    index = idx_map[val]        # Root's position in inorder

    root.right = helper(index + 1, in_right)   # Build right subtree FIRST
    root.left  = helper(in_left,   index - 1)  # Then left subtree

    return root
```

### Why Right Before Left?
This is the most subtle part of the construction. `postorder.pop()` reads elements from the end of the array. The end of postorder looks like:

```
[... left subtree ... | ... right subtree ... | root]
```

So after popping the root, the **next pop will yield the root of the right subtree**, not the left. Building `right` before `left` ensures `pop()` stays synchronized with the correct subtree at each recursive level.

---

## Walkthrough Example

```
inorder   = [9, 3, 15, 20, 7]
postorder = [9, 15, 7, 20, 3]
```

| Call | `pop()` | Root | Inorder window | Next: build |
|------|---------|------|----------------|-------------|
| `helper(0, 4)` | `3` | 3 | `[9, 3, 15, 20, 7]` | right `(2,4)`, left `(0,0)` |
| `helper(2, 4)` | `20` | 20 | `[15, 20, 7]` | right `(4,4)`, left `(2,2)` |
| `helper(4, 4)` | `7` | 7 | `[7]` | base cases |
| `helper(2, 2)` | `15` | 15 | `[15]` | base cases |
| `helper(0, 0)` | `9` | 9 | `[9]` | base cases |

**Resulting tree:**
```
        3
       / \
      9   20
         /  \
        15    7
```

## Complexity

| | Complexity | Reason |
|---|---|---|
| **Time** | O(n) | Each node is processed once; hashmap gives O(1) lookups |
| **Space** | O(n) | Hashmap stores n entries; recursion stack is O(h), worst case O(n) |

---

## Key Takeaways

- **Postorder's `pop()` naturally sequences root discovery** — no separate index tracking needed on the postorder side.
- **Inorder bounds, not array slices** — passing `in_left`/`in_right` avoids O(n) slicing at every recursive call.
- **Right subtree before left** is the non-obvious but critical ordering that keeps `pop()` in sync with the recursive structure.
- The pattern generalizes: a similar approach (with mirrored ordering) solves the **preorder + inorder** variant.
