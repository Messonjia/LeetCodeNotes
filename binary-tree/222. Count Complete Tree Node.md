# Count Nodes in a Complete Binary Tree

## Problem Summary

Given the root of a **complete binary tree**, return the total number of nodes. The algorithm must run in **less than O(n)** time complexity.

> A complete binary tree has every level fully filled except possibly the last, which is filled from left to right. The last level can have between `1` and `2^h` nodes, where `h` is the height.

---

## The Solution

```python
class Solution:
    def countNodes(self, root: Optional[TreeNode]) -> int:
        return self.getNodeInSubTree(root)

    def getNodeInSubTree(self, cur):
        if not cur:
            return 0
        leftNode = self.getNodeInSubTree(cur.left)
        rightNode = self.getNodeInSubTree(cur.right)
        treeNum = leftNode + rightNode + 1
        return treeNum
```

---

## Why Post-Order Traversal Is the Key

### What Is Post-Order?

Post-order traversal visits nodes in the order: **Left → Right → Root**.

That means before a node does *any* work, **both of its children have already reported their results**. This is exactly the contract we need here: a node cannot know how many nodes are in its subtree until it knows how many are in its left and right subtrees first.

### Why It Fits Perfectly Here

The count of nodes in a subtree rooted at `cur` is:

```
count(cur) = count(cur.left) + count(cur.right) + 1
```

The `+ 1` accounts for `cur` itself. This formula is **inherently post-order** — you need the children's answers before you can compute the parent's answer. Trying to use pre-order or in-order here would be awkward or incorrect, because the parent would be "counted" before its children's subtree sizes are known.

### The Execution Flow

For a small tree:

```
        1
       / \
      2   3
     / \
    4   5
```

Post-order visits: `4 → 5 → 2 → 3 → 1`

At each step:
| Node | Left Count | Right Count | Returns |
|------|-----------|------------|---------|
| 4    | 0          | 0          | 1       |
| 5    | 0          | 0          | 1       |
| 2    | 1          | 1          | 3       |
| 3    | 0          | 0          | 1       |
| 1    | 3          | 1          | **5**   |

Each node's count **bubbles up** from the leaves to the root — a natural bottom-up aggregation that only post-order enables cleanly.

---

## Time and Space Complexity

| | Complexity | Notes |
|---|---|---|
| **Time** | O(n) | Every node is visited once |
| **Space** | O(h) = O(log n) | Recursion stack depth equals tree height |

> ⚠️ **Note:** This solution is O(n) in time. A truly sub-O(n) solution for a complete binary tree would use the property that if the leftmost and rightmost depths are equal, the subtree is a *perfect* binary tree and its count is `2^h - 1`, enabling whole subtrees to be skipped. However, the post-order structure here remains the correct mental model — the sub-O(n) optimization is layered on top of it.

---

## The Big Takeaway

> **Post-order traversal is ideal whenever a node's answer depends on aggregated information from its children.**

Counting nodes, computing subtree heights, checking balance, summing subtree values — all of these are problems where the answer "flows upward" from leaves to root. Post-order makes this flow natural and correct by guaranteeing children are fully resolved before their parent runs.
