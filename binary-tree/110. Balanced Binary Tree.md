# Balanced Binary Tree

## Problem Summary

Given the root of a binary tree, determine if it is **height-balanced**.

> A binary tree is height-balanced if, for every node in the tree, the height difference between its left and right subtrees is **no more than 1**.

---

## The Solution

```python
class Solution:
    def isBalanced(self, root: Optional[TreeNode]) -> bool:
        return self.getHeight(root) != -1

    def getHeight(self, node):
        if not node:
            return 0
        left = self.getHeight(node.left)
        right = self.getHeight(node.right)
        if left == -1 or right == -1 or abs(left - right) > 1:
            return -1
        return max(left, right) + 1
```

---

## The Clever Trick: Sentinel Value `-1`

The most elegant decision in this solution is **overloading the return value** of `getHeight`.

Normally, `getHeight` returns a non-negative integer representing the height of a subtree. But here, `-1` is repurposed as a **poison value** — a signal that somewhere in the subtree, a balance violation was already detected.

This means the function does two jobs at once:
- If it returns `>= 0` → the subtree is balanced and the value is its height
- If it returns `-1` → the subtree is **unbalanced**, and the violation propagates upward

This avoids a separate `isBalanced` check at every node, keeping the solution clean and efficient.

---

## Why Post-Order Again

Just like counting nodes, this problem requires a **bottom-up** answer — a node cannot know whether *it* is balanced until it knows the heights of both its children. That makes post-order the natural traversal:

```
Left subtree height → Right subtree height → Check balance at current node
```

The failure signal also propagates post-order: once a `-1` is returned anywhere in the tree, every ancestor immediately short-circuits and passes `-1` further up without doing any extra work.

---

## The Execution Flow

For this tree:

```
        1
       / \
      2   3
     / \
    4   5
         \
          6
```

Post-order visits: `4 → 6 → 5 → 2 → 3 → 1`

| Node | Left Height | Right Height | `abs(L-R)` | Returns |
|------|------------|-------------|-----------|---------|
| 4    | 0          | 0           | 0         | 1       |
| 6    | 0          | 0           | 0         | 1       |
| 5    | 0          | 1           | 1         | 2       |
| 2    | 1          | 2           | 1         | 3       |
| 3    | 0          | 0           | 0         | 1       |
| 1    | 3          | 1           | **2** ❌  | **-1**  |

Node `1` detects the imbalance and returns `-1`, which bubbles all the way up to `isBalanced`, returning `False`.

---

## What Makes This Better Than the Naive Approach

A naive approach might check `isBalanced` at every node and separately compute height — resulting in **O(n log n)** time because heights are recomputed redundantly.

This solution computes height and checks balance **in the same pass**, achieving:

| | Complexity | Notes |
|---|---|---|
| **Time** | O(n) | Each node visited exactly once |
| **Space** | O(h) = O(log n) | Recursion stack, O(n) worst case for skewed trees |

The `-1` sentinel is what makes this single-pass possible — it carries the "already broken" state upward without any extra data structure.

---

## The Big Takeaway

> **Post-order traversal is the right tool whenever a node's validity depends on aggregated properties of its children — and when a failure deep in the tree should immediately invalidate everything above it.**

The sentinel value pattern (`-1` as a poison) is a reusable technique worth memorizing. You'll see it again in problems like finding the diameter of a binary tree, checking if a tree is a valid BST by height, or any problem where you want to **combine result computation with early termination** in a single recursive pass.
