# LeetCode — Maximum Binary Tree

## Problem

Given an integer array `nums` with **no duplicates**, build the **maximum binary tree** using this recursive rule:

1. The **root** is the **maximum value** in the current subarray.
2. The **left subtree** is built from everything to the **left** of the maximum.
3. The **right subtree** is built from everything to the **right** of the maximum.

Return the root of the constructed tree.

---

## Core Observation (How the Solution Was Constructed)

The problem definition *is* the algorithm — it directly describes a recursive structure:

> "Find the max, split the array, recurse on both halves."

This means no clever traversal property needs to be discovered (unlike the inorder/postorder problem). The challenge is in **translating the recursive definition cleanly into code**, and understanding the base cases.

---

## Construction Strategy

### Base Case
```python
if len(nums) == 1:
    return TreeNode(nums[0])
```
A single-element array has no left or right subtree — just return a leaf node.

### Find the Maximum and Its Index
```python
for i in range(len(nums)):
    if nums[i] > max_value:
        max_value = nums[i]
        max_value_idx = i

node.val = max_value
```
A linear scan finds both the max value (to set as root) and its index (to define the split point). This is O(n) per level.

### Conditionally Build Left and Right Subtrees
```python
if max_value_idx > 0:
    node.left = self.constructMaximumBinaryTree(nums[:max_value_idx])

if max_value_idx < len(nums) - 1:
    node.right = self.constructMaximumBinaryTree(nums[max_value_idx + 1:])
```
The conditions guard against passing an **empty array** into the recursion:
- `max_value_idx > 0` — there are elements to the left
- `max_value_idx < len(nums) - 1` — there are elements to the right

If either condition is false, that child simply stays `None`.

---

## Walkthrough Example

```
nums = [3, 2, 1, 6, 0, 5]
```

| Call | Subarray | Max | Left subarray | Right subarray |
|------|----------|-----|---------------|----------------|
| root | `[3,2,1,6,0,5]` | **6** (idx 3) | `[3,2,1]` | `[0,5]` |
| left of 6 | `[3,2,1]` | **3** (idx 0) | `[]` | `[2,1]` |
| right of 3 | `[2,1]` | **2** (idx 0) | `[]` | `[1]` |
| right of 2 | `[1]` | **1** | leaf | leaf |
| right of 6 | `[0,5]` | **5** (idx 1) | `[0]` | `[]` |
| left of 5 | `[0]` | **0** | leaf | leaf |

**Resulting tree:**
```
            6
           / \
          3   5
           \ /
           2 0
            \
             1
```

---

## Complexity

| | Complexity | Reason |
|---|---|---|
| **Time** | O(n²) worst case | Each level scans its subarray linearly; a sorted input produces a skewed tree with n levels |
| **Time** | O(n log n) average | Balanced splits when max tends toward the middle |
| **Space** | O(n) | Array slices at each level + recursion stack depth O(n) worst case |

> **Note:** The array slicing (`nums[:idx]`, `nums[idx+1:]`) creates new lists at every call. This contributes to space usage but does not change the time complexity class.

---

## Comparison to the Inorder + Postorder Problem

| | Max Binary Tree | Build from Traversals |
|---|---|---|
| **Split signal** | Index of maximum value | Root from `postorder.pop()` |
| **Recursion direction** | Left then right (order doesn't matter here) | Right then left (must match pop order) |
| **Lookup optimization** | Linear scan each call | Hashmap for O(1) inorder lookup |
| **Subarray passing** | Explicit slices | Index bounds only (no slicing) |

A natural optimization for this problem — mirroring the traversal solution — would be to pass index bounds (`left`, `right`) instead of slicing, and use a sparse table or segment tree to answer range-maximum queries in O(1), bringing overall time to O(n log n) or O(n).

---

## Key Takeaways

- The **problem statement directly encodes the recursive algorithm** — the main work is careful base case handling and guarding empty subarrays.
- The **split point is the max index**, not a fixed midpoint — this is what distinguishes it from standard divide-and-conquer like merge sort.
- **Worst case is O(n²)** when the array is sorted, producing a fully right-skewed (or left-skewed) tree with n recursive levels, each scanning a shrinking subarray.
- The guard conditions (`if max_value_idx > 0`) are essential — without them, an empty slice would bypass the base case and cause incorrect behavior.
