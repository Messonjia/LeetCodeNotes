# LC 1005 · Maximize Sum of Array After K Negations

## Problem Statement

Given an integer array `nums` and an integer `k`, you must flip the sign of exactly `k` elements (the same index may be chosen multiple times). Return the **largest possible sum** after exactly `k` flips.

**Constraints:**
- `1 <= nums.length <= 10^4`
- `-100 <= nums[i] <= 100`
- `1 <= k <= 10^4`
- Note: `k` can be **much larger** than `len(nums)` — this is a key edge case.

**Examples:**
```
Input: nums = [4,2,3],        k = 1  →  Output: 5
Input: nums = [3,-1,0,2],     k = 3  →  Output: 6
Input: nums = [2,-3,-1,5,-4], k = 2  →  Output: 13
```

---

## Key Insight — Two-Phase Greedy

### Why greedy works here
Flipping a negative into a positive always increases the sum. Among negatives, flipping the one with the **largest absolute value** gives the biggest gain. Once no negatives remain, any extra flip hurts the sum — so you minimize damage by flipping the **smallest absolute value** element, and only when `k` is odd (even leftover flips cancel out).

### Phase 1 — Eliminate negatives (most damaging first)
Sort by absolute value **descending**. Iterate and flip each negative you encounter, decrementing `k` each time. Stop when you run out of negatives or `k` reaches 0.

### Phase 2 — Burn off leftover flips
After Phase 1, if `k > 0`:
- `k` is **even** → do nothing (pairs of flips cancel perfectly)
- `k` is **odd** → flip the element with the **smallest absolute value** exactly once (last element after the descending sort)

## Solution (Annotated)

```python
class Solution:
    def largestSumAfterKNegations(self, nums: List[int], k: int) -> int:
        # Sort descending by absolute value so we process the most impactful
        # negatives first, and the smallest |value| element ends up last.
        nums.sort(key=abs, reverse=True)

        # Phase 1: flip negatives from largest |value| to smallest
        for i in range(len(nums)):
            if nums[i] < 0 and k > 0:
                nums[i] = -nums[i]
                k -= 1

        # Phase 2: leftover flips — only costs anything if k is odd
        if k % 2 == 1:
            nums[-1] = -nums[-1]   # nums[-1] has the smallest |value| after sort

        return sum(nums)
```

**Why sort by `abs` descending?**
It guarantees two things at once:
1. Negatives with the largest magnitude are flipped first (maximum gain per flip).
2. After all negatives are gone, `nums[-1]` is automatically the element with the smallest absolute value — the cheapest possible element to sacrifice in Phase 2.

**Why `k % 2 == 1` and not `k > 0`?**
If `k` remaining flips are even, flipping any element an even number of times returns it to its original value — net zero. Only an odd remainder forces a permanent sign change.

## Trace Table — `nums = [2, -3, -1, 5, -4]`, `k = 2`

**After sort by |value| descending:** `[5, -4, -3, 2, -1]`

| i | nums[i] | Action              | k after | Array state            |
|---|---------|---------------------|---------|------------------------|
| 0 | 5       | positive, skip      | 2       | [5, -4, -3, 2, -1]     |
| 1 | -4      | flip → 4            | 1       | [5, **4**, -3, 2, -1]  |
| 2 | -3      | flip → 3            | 0       | [5, 4, **3**, 2, -1]   |
| 3 | 2       | k=0, stop           | 0       | [5, 4, 3, 2, -1]       |

**Phase 2:** k=0, even → no flip.  
**Sum:** 5 + 4 + 3 + 2 + (−1) = **13** ✅

## Trace Table — `nums = [3, -1, 0, 2]`, `k = 3`

**After sort by |value| descending:** `[3, 2, -1, 0]`

| i | nums[i] | Action         | k after | Array state          |
|---|---------|----------------|---------|----------------------|
| 0 | 3       | positive, skip | 3       | [3, 2, -1, 0]        |
| 1 | 2       | positive, skip | 3       | [3, 2, -1, 0]        |
| 2 | -1      | flip → 1       | 2       | [3, 2, **1**, 0]     |
| 3 | 0       | positive, skip | 2       | [3, 2, 1, 0]         |

**Phase 2:** k=2, even → no flip.  
**Sum:** 3 + 2 + 1 + 0 = **6** ✅

## Edge Case — `k` Much Larger Than `len(nums)`

**`nums = [4, 2, 3]`, `k = 1`**  
After sort: `[4, 3, 2]` — all positive, Phase 1 does nothing.  
Phase 2: k=1 (odd) → flip `nums[-1]` = 2 → −2.  
Sum: 4 + 3 + (−2) = **5** ✅

**`nums = [1]`, `k = 10000`**  
After sort: `[1]` — Phase 1 does nothing.  
Phase 2: k=10000 (even) → no flip.  
Sum: **1** ✅ (10000 pairs of flips all cancel)

**`nums = [1]`, `k = 9999`**  
Phase 2: k=9999 (odd) → flip → −1.  
Sum: **−1** ✅ (forced — must use all k flips)

## Common Pitfalls

| Mistake | Why it fails | Fix |
|---------|-------------|-----|
| Sorting ascending by value (not by `abs`) | Flips the smallest value first (e.g. −1 before −100), missing big gains | Use `key=abs, reverse=True` |
| Using `k > 0` instead of `k % 2 == 1` in Phase 2 | Flips the smallest element even when remaining k is even, unnecessarily reducing sum | Check parity: only flip if `k % 2 == 1` |
| Not sorting at all | Phase 2 may flip a large-value element instead of the smallest | Always sort before Phase 2 |
| Assuming `k <= len(nums)` | Misses cases where all positives are present but k still has remainder | Handle Phase 2 regardless of array length |

---

## Key Takeaways

1. **Sort by absolute value descending** — this single sort serves double duty: biggest-impact negatives first, cheapest sacrifice element last.
2. **Two-phase greedy:** eliminate negatives → handle remainder parity.
3. **Parity is the crux.** Even leftover flips cancel; only odd remainder forces a real cost.
4. **`k` can dwarf `len(nums)`** — never assume all flips are "used up" during Phase 1.
