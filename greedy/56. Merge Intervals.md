# LC 56 · Merge Intervals

## Problem

Given an array of `intervals` where `intervals[i] = [starti, endi]`, merge all overlapping intervals and return an array of the non-overlapping intervals that cover all the intervals in the input.

**Example 1:**
```
Input:  intervals = [[1,3],[2,6],[8,10],[15,18]]
Output: [[1,6],[8,10],[15,18]]
```

**Example 2:**
```
Input:  intervals = [[1,4],[4,5]]
Output: [[1,5]]
```

---

## Core Insight

After sorting by start time, any overlapping interval must have its start ≤ the current window's end. So we only need to track the **last merged interval** in `result`. For each new interval:
- **Overlap** → extend the last interval's end to `max(last.end, current.end)`
- **No overlap** → append the new interval as-is

Sorting guarantees we never need to look backwards.

---

## Annotated Solution

```python
class Solution:
    def merge(self, intervals: List[List[int]]) -> List[List[int]]:

        result = []
        if len(intervals) == 0:          # edge case: empty input
            return result

        intervals.sort(key=lambda x: x[0])  # sort by start time — O(n log n)
        result.append(intervals[0])          # seed result with the first interval

        for i in range(1, len(intervals)):
            # result[-1] is always the last merged interval (the active window)
            if result[-1][1] >= intervals[i][0]:   # overlap condition
                result[-1][1] = max(intervals[i][1], result[-1][1])  # extend end
            else:
                result.append(intervals[i])        # no overlap → new partition

        return result
```

---

## Why `max()` Is Necessary

Without `max`, a fully contained interval like `[1,10]` followed by `[2,5]` would shrink the end from 10 to 5:

```
result[-1] = [1, 10],  intervals[i] = [2, 5]
# Wrong:  result[-1][1] = 5    ← shrinks the window!
# Right:  result[-1][1] = max(5, 10) = 10
```

## Complexity

|          |                                              |
|----------|----------------------------------------------|
| **Time** | O(n log n) — dominated by the sort          |
| **Space** | O(n) — result list in the worst case (no overlaps) |

---

## Key Takeaways

- **Sort first:** Sorting by start time is the key prerequisite. Without it, you'd need to compare every pair — O(n²).
- **In-place merge via `result[-1]`:** Instead of building a new interval object, directly mutate the last element of `result`. This is both space-efficient and clean.
- **Pattern — Greedy Interval Merging:** Extend the active window as far right as needed, then close it when no overlap exists. Same family as:
  - LC 763 · Partition Labels
  - LC 57 · Insert Interval
  - LC 435 · Non-overlapping Intervals

---

## Common Pitfalls

- **Forgetting `max()` on the end:** A fully contained interval (e.g., `[2,5]` inside `[1,10]`) would incorrectly shrink the merged end.
- **Sorting by end instead of start:** The overlap check `result[-1][1] >= intervals[i][0]` only works correctly when sorted by start.
- **Mutating the input:** `intervals.sort()` modifies the original list in place. If the caller needs the original, copy it first with `intervals = sorted(intervals, key=lambda x: x[0])`.
