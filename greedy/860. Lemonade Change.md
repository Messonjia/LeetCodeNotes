# LC 860 · Lemonade Change

## Problem Statement

Each lemonade costs **$5**. Customers pay with `$5`, `$10`, or `$20` bills. You start with no change.

Given `bills`, return `True` if you can give every customer exact change, `False` otherwise.

---

## Core Insight: Greedy Change Dispensing

Track only two denominations in hand: `bill5` and `bill10`. You never need to track `bill20` because $20 bills can never be used as change (the most you'd ever give back is $15).

The key greedy decision appears when a customer pays $20 — you have two ways to make $15 change:

- **Option A:** one $10 + one $5
- **Option B:** three $5s

**Always prefer Option A.** The $10 bill is less flexible than the $5 — it can only be used as change for $20. The $5 bill is the most versatile denomination and can be used as change for both $10 and $20 payments. Spending $10s first preserves your $5s for future use.

---

## Solution

```python
from typing import List

class Solution:
    def lemonadeChange(self, bills: List[int]) -> bool:
        bill5 = 0
        bill10 = 0

        for bill in bills:
            if bill == 5:
                bill5 += 1                   # no change needed, pocket the $5

            elif bill == 10:
                if bill5 <= 0:
                    return False             # can't make $5 change
                bill10 += 1
                bill5 -= 1

            else:                            # bill == 20
                if bill10 > 0 and bill5 > 0:
                    bill10 -= 1              # prefer $10 + $5 (saves $5s)
                    bill5 -= 1
                elif bill5 >= 3:
                    bill5 -= 3               # fallback: three $5s
                else:
                    return False

        return True
```

> Note: `bill20` is not tracked because $20 bills are never useful as change.

---

## Why Prefer $10 + $5 Over Three $5s?

| Denomination | Can be used as change for... |
|---|---|
| $5 | $10 payments, $20 payments |
| $10 | $20 payments only |

Because $5 bills are more versatile, spending a $10 (less flexible) first conserves $5s (more flexible) for future transactions. This greedy choice is provably optimal — if you spend three $5s when you had a $10 available, you may run out of $5s later and fail a transaction you could have otherwise handled.

## Common Pitfalls

**1. Tracking `bill20` — unnecessary.**  
You never give a $20 bill as change, so tracking it adds nothing. The moment you accept a $20 bill from a customer, it's permanently in your pocket with no future utility.

**2. Using `if` instead of `elif`.**  
The original code used chained `if` statements, which works correctly here because the conditions are mutually exclusive (a bill is exactly one denomination). However, `elif` is clearer and avoids accidentally evaluating all branches every iteration.

**3. Greedy order for $20 — wrong priority.**  
If you check `bill5 >= 3` before `bill10 > 0 and bill5 > 0`, you may spend three $5 bills unnecessarily, depleting the most versatile denomination and failing a later transaction. Always check Option A first.

```python
# ❌ Wrong order — wastes $5s
if bill5 >= 3:
    bill5 -= 3
elif bill10 > 0 and bill5 > 0:
    bill10 -= 1; bill5 -= 1

# ✅ Correct — preserve $5s
if bill10 > 0 and bill5 > 0:
    bill10 -= 1; bill5 -= 1
elif bill5 >= 3:
    bill5 -= 3
```

---

## Key Takeaways

1. **Greedy works here** because all decisions are local and irrevocable — you can prove that always spending the least-flexible denomination first never creates a worse outcome.
2. **Only $5 and $10 matter** — $20 bills are never useful as change; tracking them is wasted state.
3. **$5 is the scarcest resource** — it's required for both $10 and $20 change. Protect it.
4. This problem is a clean example of **greedy coin-exchange**: prefer spending high-denomination change coins that have fewer future uses. The same logic appears in classic coin-change problems and scheduling problems with resource constraints.
