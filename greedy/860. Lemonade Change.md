# LC 45 · Jump Game II

## Problem Statement

Given a 0-indexed integer array `nums` of length `n`, you start at index 0.  
Each `nums[i]` is the **maximum** forward jump length from index `i`.  
Return the **minimum number of jumps** to reach index `n - 1`.

**Constraints:** It is guaranteed that you can always reach the last index.

**Examples:**
```
Input:  nums = [2, 3, 1, 1, 4]   →  Output: 2
Input:  nums = [2, 3, 0, 1, 4]   →  Output: 2
```

---

## Key Insight — Greedy "Jump Windows"

Think in **windows**:
- The current jump covers indices `[0 .. cur_distance]`.
- As you scan through that window you track the **farthest index reachable** from any position inside it (`next_distance`).
- The moment you exhaust the current window (i.e., `i == cur_distance`), you **must** jump — increment the counter and extend the window to `next_distance`.

You never need to decide *which* index to jump from; just track how far you can reach within each window.

---

## Solution (Annotated)

```python
class Solution:
    def jump(self, nums: List[int]) -> int:
        cur_distance  = 0   # right edge of the current jump's reachable window
        result        = 0   # number of jumps taken so far
        next_distance = 0   # farthest index reachable within the current window

        for i in range(len(nums)):
            # Always update the farthest index reachable from position i
            next_distance = max(nums[i] + i, next_distance)

            # We've reached the end of the current window — must jump now
            if i == cur_distance:
                result        += 1
                cur_distance   = next_distance   # extend window

                # Early exit: if the new window already covers the last index
                if next_distance >= len(nums) - 1:
                    break

        return result
```

**Why `i == cur_distance` triggers a jump:**  
Every index up to `cur_distance` belongs to "jump N". When `i` hits that boundary, the next step *must* cost an extra jump, so we commit, extend the window, and continue.

**Why `i < len(nums) - 1` is not needed in the loop condition:**  
The early-exit `break` fires as soon as the reachable window covers the last index, so you never process indices past it unnecessarily.

## Complexity Analysis

| Dimension | Value  | Reason                                      |
|-----------|--------|---------------------------------------------|
| Time      | O(n)   | Single left-to-right scan of `nums`         |
| Space     | O(1)   | Three integer variables, no extra structure |

---

## Common Pitfalls

| Mistake | Why it fails | Fix |
|---------|-------------|-----|
| Jumping every time `next_distance` improves | Over-counts jumps; a jump is only needed at the window boundary | Increment `result` only when `i == cur_distance` |
| Forgetting early exit | Processes the last index and triggers an extra jump | Add `if next_distance >= len(nums) - 1: break` inside the `if` block |
| Using `range(len(nums) - 1)` | Misses the update for the last index in edge cases | Use `range(len(nums))` and let early exit handle termination |
| BFS with an explicit queue | Correct but O(n) space; the greedy approach needs O(1) | Use the window-based greedy instead |

---

## Key Takeaways

1. **Greedy by window, not by position.** You don't need to know *where* the optimal jump lands — just how far the current window reaches.
2. **Jump count = number of window expansions.** Each `i == cur_distance` event corresponds to exactly one forced jump.
3. **Early exit is safe and important.** Once `next_distance >= n - 1`, the answer is already in `result`; no further scanning needed.
4. **Compare with LC 55 (Jump Game I).** That problem only asks *can* you reach the end (boolean); LC 45 asks *how many* jumps (greedy count). Both use the same reachability scan, but LC 45 layers the window-jump counter on top.
