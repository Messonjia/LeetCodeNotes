# LeetCode 455 · Assign Cookies

## Problem Statement

You are a parent who wants to give children cookies. Each child can receive **at most one cookie**.

- `g[i]` = greed factor of child `i` — the **minimum** cookie size that will content them
- `s[j]` = size of cookie `j`
- A child `i` is content if assigned a cookie `j` where `s[j] >= g[i]`

**Goal**: maximize the number of content children.

---

## Core Idea: Greedy

Match the **greediest child** to the **largest available cookie** first. If the largest cookie can satisfy the greediest child, great — assign it. If not, no other cookie can satisfy that child either, so skip them and try the next greediest child with the same largest cookie.

This greedy strategy works because:
- Wasting a large cookie on an easy-to-satisfy child is suboptimal
- If a cookie can't satisfy the most demanding child, it might still satisfy a less demanding one

---

## Algorithm

1. Sort both `g` (greed) and `s` (cookie sizes) in ascending order
2. Use two pointers starting from the **end** of both arrays (largest values)
3. For each child from greediest to least:
   - If the largest remaining cookie satisfies them → assign it, move both pointers left
   - If not → skip this child (no cookie can satisfy them), move child pointer left only
4. Return the count of assignments

```python
g.sort()
s.sort()
result = 0
index = len(s) - 1          # pointer for largest remaining cookie

for i in range(len(g)-1, -1, -1):   # iterate children greediest first
    if index >= 0 and g[i] <= s[index]:
        result += 1
        index -= 1           # cookie used, move to next largest

return result
```

## Why Sort in Ascending Order?

Sorting ascending lets us use `len - 1` as the starting index to access the **largest** element. Iterating from right to left on both arrays naturally processes largest-to-smallest without needing to reverse.

---

## Common Bug: Missing `index >= 0` Guard

### What happens without it?

```python
# ❌ Missing guard
if g[i] <= s[index]:
```

When all cookies are used up, `index` becomes `-1`. In Python, `s[-1]` is **valid** — it wraps around and accesses the **last element** of the list instead of raising an error.

```python
s = [1, 1]
index = -1
s[index]   # → s[-1] → 1  (silently wrong!)
```

So the solver would keep comparing against `s[-1]` as if there were still cookies left, potentially over-counting `result`.

### The fix

```python
# ✅ With guard
if index >= 0 and g[i] <= s[index]:
```

`index >= 0` short-circuits the check — if there are no cookies left, we immediately skip without ever touching `s[index]`. This is especially important in Python where negative indexing never throws an `IndexError`.

---

## FAQ

### Q: Why iterate children from greediest to least, not least to most?

If we started from the least greedy child, we'd waste large cookies on easy-to-satisfy children. Greedy-first ensures we only use a large cookie when necessary, leaving smaller cookies available for less demanding children.

### Q: Why does `index` only decrement on a successful assignment?

Because a cookie that fails to satisfy the greediest child might still satisfy a less greedy one. We never discard a cookie just because it couldn't satisfy one child — we keep it and try it against the next child down.

| | Value | Reason |
|---|---|---|
| Time | O(n log n + m log m) | Sorting both arrays; `n = len(g)`, `m = len(s)` |
| Space | O(1) | Only two pointers and a counter; sorting in-place |

## Complete Solution

```python
class Solution:
    def findContentChildren(self, g: List[int], s: List[int]) -> int:
        g.sort()
        s.sort()
        result = 0
        index = len(s) - 1

        for i in range(len(g) - 1, -1, -1):
            if index >= 0 and g[i] <= s[index]:
                result += 1
                index -= 1

        return result
```

---

## Key Takeaways

- **Greedy match**: always try to satisfy the greediest child first with the largest available cookie
- **Sort both arrays** so largest values are at the end, enabling right-to-left traversal
- **`index` only moves on success** — a cookie that fails one child may still work for another
- **Always guard `index >= 0`** before accessing `s[index]` — in Python, negative indices silently wrap around instead of raising an error, which can cause subtle over-counting bugs
