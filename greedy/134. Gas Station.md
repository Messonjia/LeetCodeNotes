# LC 134 · Gas Station

## Problem Statement

There are `n` gas stations along a **circular** route. `gas[i]` is the fuel available at station `i`; `cost[i]` is the fuel needed to travel from station `i` to station `i+1`.

You start with an empty tank. Return the index of the starting station from which you can complete the full circuit clockwise, or `-1` if no such station exists.

**Guarantee:** If a solution exists, it is **unique**.

**Examples:**
```
Input: gas  = [1,2,3,4,5]
       cost = [3,4,5,1,2]
Output: 3

Input: gas  = [2,3,4]
       cost = [3,4,3]
Output: -1
```

---

## Key Insight — Two Independent Observations

### Observation 1 — Global feasibility
Define `net[i] = gas[i] - cost[i]`.  
If `sum(net) < 0`, total fuel in the entire circuit is insufficient — return `-1` immediately.  
If `sum(net) >= 0`, a valid start **must** exist (guaranteed unique).

### Observation 2 — Greedy start detection
Scan left-to-right while maintaining a running balance `current_sum`.  
Whenever `current_sum` drops below 0, every station from the current `start` up to index `i` has been proven unable to carry us through — **abandon the window** and set `start = i + 1`, resetting `current_sum = 0`.

**Why this works:** If starting at `s` leads to a deficit at station `i`, then starting at any station between `s` and `i` also fails (they inherit the same deficit plus their own shortfall). The only viable candidate must lie **after** `i`.

Because we already know from Observation 1 that a solution exists, whatever `start` survives to the end of the scan is the answer — no second pass needed.

## Solution (Annotated)

```python
class Solution:
    def canCompleteCircuit(self, gas: List[int], cost: List[int]) -> int:
        current_sum = 0   # running balance from the current candidate start
        total_sum   = 0   # sum of all net values — determines global feasibility
        start       = 0   # current candidate starting index

        for i in range(len(gas)):
            current_sum += gas[i] - cost[i]
            total_sum   += gas[i] - cost[i]

            # Cannot reach the next station from the current start window
            if current_sum < 0:
                start       = i + 1   # discard everything up to i, try next
                current_sum = 0       # reset local balance for the new candidate

        # Global check: if total net fuel is negative, no solution exists
        if total_sum < 0:
            return -1

        return start   # the last surviving candidate is the unique answer
```

**Why `total_sum` is checked AFTER the loop (not before):**  
We compute `total_sum` incrementally in the same pass that finds `start`, saving a separate O(n) scan. The global check only needs to happen once at the end.

**Why no second verification pass is needed:**  
The problem guarantees uniqueness. Once `total_sum >= 0`, the greedy `start` is mathematically proven to be the only valid answer — no need to simulate the full circuit from `start`.

---

## Common Pitfalls

| Mistake | Why it fails | Fix |
|---------|-------------|-----|
| Checking `total_sum < 0` **before** the loop with a separate pass | Correct but wastes an extra O(n) scan | Accumulate `total_sum` inside the same loop |
| Resetting `total_sum` when `current_sum < 0` | Corrupts the global feasibility check | Only reset `current_sum`; always keep `total_sum` accumulating |
| Doing a second full simulation from `start` to verify | Unnecessary given the uniqueness guarantee | Trust the greedy — if `total_sum >= 0`, `start` is the answer |
| Setting `start = i` instead of `start = i + 1` | Includes the station that just caused a deficit | Station `i` itself failed; next candidate begins at `i + 1` |
| Using `current_sum <= 0` as the reset condition | Resets when balance is exactly 0 (still valid) | Use strict `< 0`; zero means we just barely make it to the next station |

---

## Key Takeaways

1. **Two variables, one pass.** `total_sum` answers *"is a solution possible?"*; `current_sum` answers *"where does it start?"* — both computed in a single O(n) scan.
2. **Greedy reset is provably optimal.** If any station in `[start..i]` could complete the circuit, the prefix sum from that station would still be non-negative at `i`. A drop below 0 invalidates the entire prefix — not just the current `start`.
3. **Uniqueness eliminates verification.** The problem guarantees at most one solution. Once `total_sum >= 0`, whatever `start` remains is the answer — no simulation needed.
4. **`net[i] = gas[i] - cost[i]`** is the core transformation. Every operation — global sum, local running balance — reduces to accumulating these net values.
