# LC 53 · Maximum Subarray

## Problem Statement

Given an integer array `nums`, find the **subarray with the largest sum** and return its sum.

**Example:**
```
Input:  nums = [-2, 1, -3, 4, -1, 2, 1, -5, 4]
Output: 6
Explanation: The subarray [4, -1, 2, 1] has the largest sum 6.
```

**Constraints:**
- `1 <= nums.length <= 10^5`
- `-10^4 <= nums[i] <= 10^4`

**Follow-up:** Can you solve it in O(n) time?

---

## Greedy Insight — Why It Works Fast

This is **Kadane's Algorithm**, a classic greedy approach.

### The Core Question

If we're building a subarray left to right, when should we cut our losses and start fresh?

**Answer:** The moment our running sum `count` drops to **≤ 0**, it becomes a liability. Any positive number following a non-positive prefix would be better off starting alone. So we reset `count = 0` — greedy discard.

```
nums:  [-2,  1, -3,  4, -1,  2,  1, -5,  4]
count:  -2→0  1  -2→0  4   3   5   6   1   5
              ↑        ↑
           kept      kept (best window starts here)
```

### Why Greedy Is Safe Here

Suppose we have prefix `[..., -5]` with a cumulative sum of `-3`. Any element `x` that follows has two choices:
- Start a new subarray: contributes just `x`
- Extend the prefix: contributes `x + (-3) = x - 3`

The second option is **strictly worse** for any `x`. So discarding a non-positive prefix is always correct — we never lose the optimal answer by resetting.

### The `result` Update Happens BEFORE the Reset

```python
if count > result:
    result = count   # snapshot the best BEFORE potentially resetting
if count <= 0:
    count = 0        # reset the window
```

This order matters. We update `result` first so we never miss a single-element maximum (e.g., if `nums = [-3]`, `count = -3`, `result = -3`, then we reset `count` but `result` is already saved).

## Annotated Solution

```python
class Solution:
    def maxSubArray(self, nums: List[int]) -> int:
        result = float('-inf')   # 初始化为负无穷，以处理全负数组的情况
        count = 0                # 当前滑动窗口的累计和

        for i in range(len(nums)):
            count += nums[i]         # 将当前元素加入窗口

            if count > result:
                result = count       # 贪心：随时记录最大值

            if count <= 0:
                count = 0            # 贪心：当前前缀和为负，丢弃，重新开始
                                     # 注意：必须先更新 result，再重置 count

        return result
```

> **Why `float('-inf')` and not `0`?** If all elements are negative (e.g., `[-3, -1, -2]`), the answer is `-1`, not `0`. Initializing to `0` would incorrectly return `0` as if an empty subarray is valid.

## Edge Case: All Negative Numbers

`nums = [-3, -1, -2]`

| i | nums[i] | count | result | Reset? |
|---|---------|-------|--------|--------|
| 0 | -3      | -3    | -3     | yes → 0 |
| 1 | -1      | -1    | -1     | yes → 0 |
| 2 | -2      | -2    | -1     | yes → 0 |

**Result: -1** ✓ (the least-negative element is the answer — NOT 0)

This is exactly why `result = float('-inf')` is needed. If `result = 0`, the answer would be wrong.

## Complexity Analysis

| | Value | Reason |
|---|---|---|
| **Time** | O(n) | Single left-to-right pass |
| **Space** | O(1) | Only `count` and `result` variables |

Compare with brute force (all subarrays): O(n²). Divide and conquer also works in O(n log n) but this greedy beats it.

---

## Key Takeaways

1. **A non-positive prefix can only hurt future sums — discard it.** This is the core greedy insight.
2. **Update `result` BEFORE resetting `count`.** This ensures single-element max values are captured correctly.
3. **Initialize `result = float('-inf')`, not `0`**, to correctly handle all-negative arrays.
4. **`count` tracks the current window sum; `result` tracks the global best.** They serve different roles — don't confuse them.
5. This greedy is equivalent to a 1D DP where `dp[i] = max(nums[i], dp[i-1] + nums[i])`, but collapses to O(1) space.

---

## Connection to DP

Kadane's greedy is the space-optimized form of the following DP recurrence:

```
dp[i] = the maximum subarray sum ending at index i
dp[i] = max(nums[i], dp[i-1] + nums[i])
```

- If `dp[i-1] > 0`: extend the previous subarray (`dp[i-1] + nums[i]`)
- If `dp[i-1] <= 0`: start fresh (`nums[i]`)

The greedy's `count` variable IS `dp[i]`, updated in-place. The `if count <= 0: count = 0` line is exactly this recurrence.

## Common Pitfalls

| Bug | Symptom | Fix |
|---|---|---|
| `result = 0` as initial value | Returns 0 for `[-3, -1]` instead of -1 | Use `result = float('-inf')` |
| Reset `count` BEFORE updating `result` | Misses negative single-element maximums | Always update `result` first |
| Use `count < 0` instead of `count <= 0` | Window of `[5, -5, 3]` — count hits 0, should reset but doesn't, no bug here actually | `<= 0` is correct: a zero prefix contributes nothing, safe to drop |
| Confuse `count` reset with `result` reset | Resets `result = 0` on discard | Only `count` resets; `result` is never reset |
