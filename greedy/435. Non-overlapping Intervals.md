# LC 435 · Non-overlapping Intervals

## Problem Statement

Given an array of intervals `intervals[i] = [starti, endi]`, return the **minimum number of intervals to remove** so that no two remaining intervals overlap.

Intervals that touch at a single point are **not** considered overlapping. For example, `[1,2]` and `[2,3]` can coexist.

---

## Core Insight: Greedy — Keep the Shorter Tail

Sort by start time. Scan left to right. When two intervals overlap, **remove the one with the larger end value** — it extends further right and is more likely to collide with future intervals. Keep the one with the smaller end.

The mutation `intervals[i][1] = min(intervals[i-1][1], intervals[i][1])` carries that smaller end forward, so the next comparison automatically uses the tightest surviving boundary.

This is the **mirror of LC 452** (Min Arrows to Burst Balloons):
- LC 452 counts **groups** of overlapping intervals (each group = one arrow).
- LC 435 counts **removals** needed to break all overlapping groups.
- Both sort by start, both use the same shrinking-window mutation.

---

## Solution

```python
from typing import List

class Solution:
    def eraseOverlapIntervals(self, intervals: List[List[int]]) -> int:
        if not intervals:
            return 0

        intervals.sort(key=lambda x: x[0])   # sort by start
        count = 0

        for i in range(1, len(intervals)):
            if intervals[i - 1][1] > intervals[i][0]:   # overlap (strictly greater)
                count += 1
                # keep the interval with the smaller end (discard the longer one)
                intervals[i][1] = min(intervals[i - 1][1], intervals[i][1])

        return count
```

---

## Why `min()` — and Why Mutate `intervals[i][1]`?

When two intervals overlap, we logically remove the one with the **larger** right end and keep the one with the smaller right end. But rather than physically deleting from the array, we simply update `intervals[i][1]` to `min(prev_end, curr_end)`.

This works because:
1. The next iteration reads `intervals[i][1]` as `intervals[i-1][1]`.
2. By writing the minimum end into `intervals[i][1]`, we ensure the surviving boundary — the tightest constraint — is propagated forward automatically.

```
intervals[i-1] = [1, 5]
intervals[i]   = [2, 3]   ← overlap: 5 > 2

Two choices:
  Remove [1, 5] (keep [2, 3]) → right boundary going forward = 3  ← CORRECT
  Remove [2, 3] (keep [1, 5]) → right boundary going forward = 5  ← BAD (more future collisions)

→ intervals[i][1] = min(5, 3) = 3   ✓
```

Always keep the interval that ends earliest — it leaves the most room for future intervals.

---

## Overlap Condition: `>` not `>=`

```python
if intervals[i - 1][1] > intervals[i][0]:   # overlap
```

The condition uses **strictly greater than**. If `prev_end == curr_start`, the intervals only touch at a point — this is explicitly **not** an overlap per the problem statement. No removal needed.

| prev | curr | `prev_end > curr_start`? | overlap? |
|------|------|--------------------------|----------|
| [1, 2] | [2, 3] | 2 > 2 → False | No — touching is OK |
| [1, 3] | [2, 4] | 3 > 2 → True  | Yes — remove one |
| [1, 2] | [3, 4] | 2 > 3 → False | No — gap between them |

---

## Trace Table

**Input:** `[[1,3],[2,4],[3,5],[3,6]]`

After sort by start: `[[1,3],[2,4],[3,5],[3,6]]`

| i | prev `[s,e]` | curr `[s,e]` | `prev_end > curr_start`? | action | count |
|---|---|---|---|---|---|
| 1 | [1, **3**] | [2, 4] | 3 > 2 → Yes | remove; `intervals[1][1] = min(3,4) = 3` → curr becomes [2,**3**] | 1 |
| 2 | [2, **3**] | [3, 5] | 3 > 3 → No | no overlap (touching) | 1 |
| 3 | [3, **5**] | [3, 6] | 5 > 3 → Yes | remove; `intervals[3][1] = min(5,6) = 5` → curr becomes [3,**5**] | 2 |

**Result: 2** ✓

Remaining intervals: `[1,3]` and `[3,5]` — they only touch, so valid.

## Complexity Analysis

| | Complexity |
|---|---|
| **Time** | O(n log n) — dominated by sorting |
| **Space** | O(1) extra — sort in-place, mutate the input array |

---

## Comparison with LC 452 (Min Arrows)

| | LC 435 (this problem) | LC 452 (arrows) |
|---|---|---|
| **Goal** | Min removals to eliminate all overlaps | Min arrows to burst all balloons |
| **Overlap condition** | `prev_end > curr_start` (strict) | `prev_end < curr_start` for non-overlap (strict) |
| **Touching boundaries** | NOT an overlap — no removal | NOT a gap — one arrow bursts both |
| **What we count** | Removals (overlapping intervals) | Arrows (non-overlapping groups) |
| **Mutation** | `intervals[i][1] = min(prev, curr)` | `points[i][1] = min(prev, curr)` |
| **Relationship** | answer = n − (max non-overlapping kept) | answer = number of kept groups |

Both problems sort by start and use the same `min()` mutation. The only difference is the boundary strictness and what gets counted.

---

## Key Takeaways

1. **Greedy: always keep the interval that ends earliest.** The longer-tailed interval collides with more future intervals and should be discarded.
2. **Mutating `intervals[i][1]`** propagates the surviving right boundary forward without a separate variable — the key implementation trick shared with LC 452.
3. **Touching is not overlapping** — use `>` (not `>=`) to detect overlaps. Off-by-one here changes the answer.
4. **Equivalent formulation:** min removals = n − (max number of non-overlapping intervals you can keep). The greedy scan is counting the latter implicitly.
5. Recognizing the LC 452 twin pattern saves time on both problems — once you see the `min()` mutation in one, the other becomes trivial.
