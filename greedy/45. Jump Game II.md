# LC 55 · Jump Game

## Problem Statement

Given an integer array `nums`, you are initially positioned at index `0`. Each element `nums[i]` represents the **maximum** jump length from position `i`. Return `true` if you can reach the last index, or `false` otherwise.

**Constraints:** `1 <= nums.length <= 10^4`, `0 <= nums[i] <= 10^5`

**Examples:**
```
Input: nums = [2,3,1,1,4]  →  Output: true
Input: nums = [3,2,1,0,4]  →  Output: false
```

---

## Solution 1 — Backward Greedy (Shrink Target)

### Intuition

Instead of asking *"where can I reach from here?"*, ask *"can anything reach my current target?"* Scan right-to-left. Whenever a position can reach the current `end`, make **that position** the new target. If we can shrink the target all the way back to index `0`, a valid path exists.

### Annotated Code

```python
class Solution:
    def canJump(self, nums: List[int]) -> bool:
        i = len(nums) - 2      # Start one before the last index
        end = len(nums) - 1    # Current target we want to reach

        while i >= 0:
            if i + nums[i] >= end:   # Can position i reach the target?
                end = i              # Greedily move the target leftward
            i -= 1

        return end == 0              # True only if target reached index 0
```

### Trace Table — `nums = [2, 3, 1, 1, 4]`

| `i` | `nums[i]` | `i + nums[i]` | `end` (before) | Update? | `end` (after) |
|-----|-----------|---------------|----------------|---------|---------------|
| 3   | 1         | 4             | 4              | ✅ 4 ≥ 4 | 3             |
| 2   | 1         | 3             | 3              | ✅ 3 ≥ 3 | 2             |
| 1   | 3         | 4             | 2              | ✅ 4 ≥ 2 | 1             |
| 0   | 2         | 2             | 1              | ✅ 2 ≥ 1 | 0             |

`end == 0` → **return True** ✅

### Trace Table — `nums = [3, 2, 1, 0, 4]`

| `i` | `nums[i]` | `i + nums[i]` | `end` (before) | Update? | `end` (after) |
|-----|-----------|---------------|----------------|---------|---------------|
| 3   | 0         | 3             | 4              | ❌ 3 < 4 | 4             |
| 2   | 1         | 3             | 4              | ❌ 3 < 4 | 4             |
| 1   | 2         | 3             | 4              | ❌ 3 < 4 | 4             |
| 0   | 3         | 3             | 4              | ❌ 3 < 4 | 4             |

`end != 0` → **return False** ❌

### Why It's Greedy

The algorithm makes a **locally optimal, irrevocable decision** at each step: the moment it finds *any* index that can reach `end`, it moves the target without checking alternatives. The greedy property is safe because reachability is monotone — if you can get to `i`, and `i` can reach `end`, then reaching `end` from `i` is all that matters; you don't need to know *how* you got to `i`.

## Solution 2 — Forward Greedy (Max Coverage)

### Intuition

Maintain a running `cover`: the **farthest index reachable** from any position seen so far. Iterate forward. As long as the current index is within our reachable range (`i <= cover`), we can legitimately stand there and potentially extend coverage. If `cover` reaches or passes the last index, return `True`. If the loop exits without that happening, return `False`.

### Annotated Code

```python
class Solution:
    def canJump(self, nums: List[int]) -> bool:
        if len(nums) == 1:         # Already at the last index
            return True

        cover = 0                  # Farthest index we can currently reach
        i = 0

        while i <= cover:          # Only visit reachable positions
            cover = max(cover, i + nums[i])   # Extend max coverage
            if cover >= len(nums) - 1:        # Reached or passed the end
                return True
            i += 1

        return False               # Loop ended before reaching the last index
```

### Trace Table — `nums = [2, 3, 1, 1, 4]`

| `i` | `nums[i]` | `i + nums[i]` | `cover` (before) | `cover` (after) | Reached end? |
|-----|-----------|---------------|------------------|-----------------|--------------|
| 0   | 2         | 2             | 0                | 2               | ❌           |
| 1   | 3         | 4             | 2                | 4               | ✅ 4 ≥ 4     |

**return True** ✅

### Trace Table — `nums = [3, 2, 1, 0, 4]`

| `i` | `nums[i]` | `i + nums[i]` | `cover` (before) | `cover` (after) | Reached end? |
|-----|-----------|---------------|------------------|-----------------|--------------|
| 0   | 3         | 3             | 0                | 3               | ❌           |
| 1   | 2         | 3             | 3                | 3               | ❌           |
| 2   | 1         | 3             | 3                | 3               | ❌           |
| 3   | 0         | 3             | 3                | 3               | ❌           |

`i = 4 > cover = 3` → loop exits → **return False** ❌

### The Key Insight: `while i <= cover`

The loop condition `i <= cover` is what makes this greedy rather than a brute-force BFS. We never enqueue/queue anything — we simply refuse to visit positions outside the current reachable window. If position `i` would require us to have already jumped there from somewhere unreachable, we never reach `i` in the loop.

## Comparison: Both Solutions

| Property               | Solution 1 (Backward)         | Solution 2 (Forward)         |
|------------------------|-------------------------------|------------------------------|
| Scan direction         | Right → Left                  | Left → Right                 |
| Core variable          | `end`: target to reach        | `cover`: max reachable index |
| Greedy decision        | Move target left on first hit | Always extend max reach      |
| Early exit             | ❌ Full scan                  | ✅ Exits as soon as end reached |
| Handles `len == 1`     | Naturally (loop skipped)      | Needs explicit guard         |
| Arguably more intuitive| Backward "chain" thinking     | Forward "horizon" thinking   |

Both run in **O(n) time** and **O(1) space**.

## Complexity Analysis

| | Time | Space |
|---|---|---|
| Solution 1 | O(n) — single backward pass | O(1) |
| Solution 2 | O(n) — single forward pass, often exits early | O(1) |

---

## Key Takeaways

1. **Greedy works here** because reachability is monotone — extending the farthest-reach greedily never hurts.
2. **Both directions work.** Backward shrinks the target; forward expands coverage. Same problem, two valid perspectives.
3. **No DP needed.** A common instinct is to reach for DP (`dp[i] = can I reach i?`), but greedy is O(n) versus DP's O(n²) and is simpler.
4. **The `while i <= cover` gate** in Solution 2 implicitly prunes unreachable paths — cleaner than checking `dp[i]` explicitly.
5. **Watch the `len == 1` edge case** in Solution 2. Solution 1 handles it naturally because the loop body never executes (`i = len-2 = -1 < 0`).
