# LC 135 · Candy

## Problem Statement

There are `n` children standing in a line. Each child is assigned a rating value in the integer array `ratings`.

You must distribute candies to these children subject to:
- Each child must have **at least 1 candy**.
- Children with a **higher rating** than their immediate neighbor must receive **more candies** than that neighbor.

Return the **minimum total number of candies** needed.

---

## Core Insight: Two-Pass Greedy

A single pass cannot satisfy both the left-neighbor and right-neighbor constraints simultaneously. The key observation is:

- **Pass 1 (L → R):** Ensure every child who is rated higher than their *left* neighbor gets more candies than that neighbor.
- **Pass 2 (R → L):** Ensure every child who is rated higher than their *right* neighbor gets more candies than that neighbor — while *keeping* any candy count that was already bumped up in Pass 1.

The second pass uses `max(candies[i], candies[i+1] + 1)` to reconcile both constraints without undoing Pass 1's work.

---

## Solution

```python
from typing import List

class Solution:
    def candy(self, ratings: List[int]) -> int:
        n = len(ratings)
        candies = [1] * n              # baseline: every child gets 1

        # Pass 1: L → R — satisfy left-neighbor constraint
        for i in range(1, n):
            if ratings[i] > ratings[i - 1]:
                candies[i] = candies[i - 1] + 1   # ← was: candies[i] + 1 (bug)

        # Pass 2: R → L — satisfy right-neighbor constraint
        for i in range(n - 2, -1, -1):
            if ratings[i] > ratings[i + 1]:
                candies[i] = max(candies[i], candies[i + 1] + 1)

        return sum(candies)
```

---

## Bug Analysis

The original Pass 1 code contained a subtle bug:

```python
# ❌ Original (wrong)
candies[i] = candies[i] + 1      # always adds 1, ignores the chain

# ✅ Fixed
candies[i] = candies[i - 1] + 1  # builds on the previous child's count
```

**Why it matters:**  
Consider `ratings = [1, 2, 3]`. The candy counts must be at least `[1, 2, 3]`.  
- With `candies[i] + 1`: you get `[1, 1+1, 1+1] = [1, 2, 2]` — child 2 ties child 1, violating the constraint.  
- With `candies[i-1] + 1`: you get `[1, 1+1, 2+1] = [1, 2, 3]` — correct.

## Trace Table

**Example:** `ratings = [1, 0, 2]`

| Step | candies[0] | candies[1] | candies[2] | Notes |
|------|-----------|-----------|-----------|-------|
| Init | 1 | 1 | 1 | All children start with 1 candy |
| L→R (i=1) | 1 | 1 | 1 | ratings[1]=0 < ratings[0]=1, no change |
| L→R (i=2) | 1 | 1 | 2 | ratings[2]=2 > ratings[1]=0 → candies[2] = candies[1]+1 = 2 |
| R→L (i=1) | 1 | 1 | 2 | ratings[1]=0 < ratings[2]=2, no change |
| R→L (i=0) | 2 | 1 | 2 | ratings[0]=1 > ratings[1]=0 → candies[0] = max(1, 1+1) = 2 |
| **Sum** | **2** | **1** | **2** | **Total = 5** |

| L→R | 1 | 2 | 3 | Ascending chain: each child gets +1 more than the previous |

## Why `max()` in Pass 2?

Pass 2 uses `max(candies[i], candies[i+1] + 1)` rather than a direct assignment for a critical reason:

- Pass 1 may have already given `candies[i]` a high value because of a long ascending run on the *left*.
- Pass 2 must honor the *right*-neighbor constraint without destroying that.
- `max()` picks whichever constraint demands more — satisfying both at once.

**Example where this matters:** `ratings = [1, 3, 2]`
- After L→R: `[1, 2, 1]` (child 1 rated higher than child 0, gets 2)
- R→L at i=1: `max(2, 1+1) = max(2, 2) = 2` — no change needed; both constraints already satisfied.
- Final: `[1, 2, 1]`, sum = **4**

---

## Key Takeaways

1. **Two-pass greedy is the classic pattern** for problems where a local constraint must be satisfied from two directions at once.
2. **Pass 1 builds chains from the left:** `candies[i] = candies[i-1] + 1` (not `candies[i] + 1`).
3. **Pass 2 reconciles with `max()`** so that neither pass's work gets overwritten.
4. **Equal ratings between neighbors** do not trigger an increment — the constraint only applies when one rating is *strictly greater* than the other. Two equal neighbors can each have 1 candy.
5. This two-pass structure generalizes to any problem requiring "local ordering constraints from both sides" (e.g., LC 42 Trapping Rain Water uses the same left-max / right-max split).
