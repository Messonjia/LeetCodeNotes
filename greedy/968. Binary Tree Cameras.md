# LC 968 · Binary Tree Cameras

## Problem

Given the `root` of a binary tree, install cameras on tree nodes where each camera can monitor its **parent, itself, and its immediate children**. Return the **minimum number of cameras** needed to monitor all nodes.

**Example 1:**
```
Input:  root = [0,0,null,0,0]
Output: 1
```

**Example 2:**
```
Input:  root = [0,0,null,0,null,0,null,null,0]
Output: 2
```

---

## Core Insight — Greedy Bottom-Up

Place cameras as **high up** (close to root) as possible by skipping leaf nodes. A camera on a leaf's parent covers the leaf, the parent, and the grandparent — maximizing coverage per camera.

**Strategy:** Traverse bottom-up (post-order). At each node, decide based on its children's states. Local optimum (never place on a leaf) → global optimum (minimum cameras).

### Node States

| State | Meaning |
|-------|---------|
| `0` | Node is **not covered** |
| `1` | Node **has a camera** |
| `2` | Node is **covered** (by a child's camera) |

**Key assumption:** `null` nodes return `2` (treated as already covered) — this prevents cameras from being placed on leaf nodes.

---

## Annotated Solution

```python
class Solution:
    def minCameraCover(self, root: Optional[TreeNode]) -> int:
        result = [0]       # mutable counter — must be [0], NOT [] (see bugs section)
        if self.traversal(root, result) == 0:  # root itself is uncovered
            result[0] += 1                     # place one more camera at root
        return result[0]

    def traversal(self, cur: TreeNode, result: List[int]) -> int:
        if not cur:
            return 2       # null nodes are assumed covered — avoids cameras on leaves

        left = self.traversal(cur.left, result)   # post-order: left first
        right = self.traversal(cur.right, result)  # then right, then current

        # Condition 1: both children are covered → this node is NOT covered yet
        # Let the parent handle it (return 0, bubble up the responsibility)
        if left == 2 and right == 2:
            return 0

        # Condition 2: at least one child is uncovered → MUST place camera here
        # This camera covers: current node, both children, and the parent
        if left == 0 or right == 0:
            result[0] += 1
            return 1

        # Condition 3: at least one child has a camera → current node is covered
        if left == 1 or right == 1:
            return 2
```

---

## Post-Order Traversal

The traversal visits nodes in **left → right → current** order. This means by the time we decide what to do at a node, we already know the full state of both its subtrees.

```
        A          ← visited last (root)
       / \
      B   C        ← visited second
     / \
    D   E          ← visited first (leaves)
```

---

## Trace — `root = [0, 0, null, 0, 0]`

```
      A
     /
    B
   / \
  D   E
```

| Node | left | right | Condition | Action | Returns |
|------|------|-------|-----------|--------|---------|
| D    | 2 (null) | 2 (null) | 1: both covered | — | 0 (uncovered) |
| E    | 2 (null) | 2 (null) | 1: both covered | — | 0 (uncovered) |
| B    | 0    | 0     | 2: child uncovered | result[0]++ → 1 | 1 (has camera) |
| A    | 1    | 2 (null) | 3: child has camera | — | 2 (covered) |

`traversal(root)` returns `2` → root is covered → no extra camera needed.

**Output:** `result[0]` = `1` ✓

---

## Why `result = [0]` and Not `result = 0`

`traversal` is a separate method that needs to increment the camera count. Using a plain `int` won't work cleanly because integers are immutable — reassigning inside a called function doesn't affect the caller's variable.

A **list is mutable**, so `result[0] += 1` modifies the same object both functions share:

| Approach | Works? | Reason |
|----------|--------|--------|
| `result = []` | ❌ | Empty list — `result[0]` doesn't exist, crashes immediately with `IndexError` |
| `result = [0]` | ✅ | Pre-initialized to 0 — `result[0]` is ready to increment |
| `self.result = 0` | ✅ | Instance variable, no scoping issue |

## Complexity

|           |                              |
|-----------|------------------------------|
| **Time**  | O(n) — visits every node once |
| **Space** | O(h) — recursion stack, where h is tree height (O(log n) balanced, O(n) worst) |

---

## Key Takeaways

- **Greedy bottom-up:** Skip leaves, place cameras at their parents. This is the locally optimal choice that leads to the global minimum.
- **3-state return value:** Using `0/1/2` from each recursive call elegantly encodes all the information the parent needs.
- **Null → 2 trick:** Returning `2` for null nodes is what prevents cameras from landing on leaf nodes — without it, every leaf would trigger condition 2 and get a camera.
- **Post-order is mandatory:** You must know children's states before deciding on the current node. Pre-order or in-order won't work here.

---

## Common Bugs

| Bug | Error | Fix |
|-----|-------|-----|
| `result = []` instead of `result = [0]` | `IndexError: list index out of range` | Initialize as `result = [0]` |
| Returning `null` nodes as `0` instead of `2` | Cameras placed on every leaf — wrong count | `if not cur: return 2` |
| Using pre-order instead of post-order | Children's states unknown when parent is processed | Always recurse on children before processing current node |
