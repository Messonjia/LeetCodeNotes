# LC 45 ┬╖ Jump Game II

## Problem Statement

Given a 0-indexed integer array `nums` of length `n`, you start at index 0.  
Each `nums[i]` is the **maximum** forward jump length from index `i`.  
Return the **minimum number of jumps** to reach index `n - 1`.

**Constraints:** It is guaranteed that you can always reach the last index.

**Examples:**

## Problem Statement

The Fibonacci sequence is defined as:

```
F(0) = 0
F(1) = 1
F(n) = F(n-1) + F(n-2),  for n > 1
```

Given an integer `n`, return `F(n)`.

**Constraints:** `0 <= n <= 30`

---

## Solution тАФ Space-Optimized DP

Instead of storing the entire DP array, we only keep track of the **last two values** using a two-element list `dp = [prev2, prev1]`, rolling it forward each iteration.

```python
class Solution:
    def fib(self, n: int) -> int:
        if n <= 1:
            return n
        
        total = 0
        dp = [0, 1]           # dp[0] = F(i-2), dp[1] = F(i-1)

        for i in range(2, n + 1):   # iterate from F(2) up to F(n), inclusive
            total = dp[0] + dp[1]   # F(i) = F(i-2) + F(i-1)
            dp[0] = dp[1]           # slide window: old F(i-1) becomes new F(i-2)
            dp[1] = total           # slide window: F(i) becomes new F(i-1)

        return total
```

> **Note:** Avoid naming the accumulator variable `sum` тАФ it shadows Python's built-in `sum()` function.

## тЪая╕П Common Bug тАФ Wrong `range`

### Why `range(n)` is incorrect

`range(n)` generates indices `0, 1, 2, ..., n-1`, which is **n iterations starting from 0**.  
Inside the loop, `i` starts at `0`, but `dp` is already seeded with `F(0)` and `F(1)`.  
This means the loop **re-computes values that already exist** and **stops one step too early**.

| `range(n)` тАФ тЭМ | Actual meaning |
|---|---|
| `i = 0` | Tries to compute F(2) but thinks it's the "0th" step тАФ off by 2 |
| Last `i = n-1` | Loop ends after computing F(n-1), **never reaches F(n)** |

For `n = 5`, `range(5)` runs 5 times, but we only need **4 new computations** (F(2) through F(5)).  
The result returned would be `F(4) = 3` instead of the correct `F(5) = 5`. тЭМ

---

### Why `range(2, n + 1)` is correct

We seed `dp` with the known base cases `F(0) = 0` and `F(1) = 1`.  
The loop only needs to compute the **new** values starting from `F(2)`.

| Iteration | `i` | Computes |
|---|---|---|
| 1st | 2 | F(2) = F(0) + F(1) = 1 |
| 2nd | 3 | F(3) = F(1) + F(2) = 2 |
| 3rd | 4 | F(4) = F(2) + F(3) = 3 |
| 4th | 5 | F(5) = F(3) + F(4) = 5 |

- **Start at `2`** тАФ because `F(0)` and `F(1)` are already handled by the base case.  
- **End at `n + 1`** тАФ because Python's `range` is **exclusive** on the right, so `range(2, n+1)` includes `n` as the last value, ensuring we compute all the way up to `F(n)`.

## Key Insight тАФ Greedy "Jump Windows"

Think in **windows**:
- The current jump covers indices `[0 .. cur_distance]`.
- As you scan through that window you track the **farthest index reachable** from any position inside it (`next_distance`).
- The moment you exhaust the current window (i.e., `i == cur_distance`), you **must** jump тАФ increment the counter and extend the window to `next_distance`.

You never need to decide *which* index to jump from; just track how far you can reach within each window.

---

## Solution (Annotated)

```python
class Solution:
    def jump(self, nums: List[int]) -> int:
        cur_distance  = 0   # right edge of the current jump's reachable window
        result        = 0   # number of jumps taken so far
        next_distance = 0   # farthest index reachable within the current window

        for i in range(len(nums)):
            # Always update the farthest index reachable from position i
            next_distance = max(nums[i] + i, next_distance)

            # We've reached the end of the current window тАФ must jump now
            if i == cur_distance:
                result        += 1
                cur_distance   = next_distance   # extend window

                # Early exit: if the new window already covers the last index
                if next_distance >= len(nums) - 1:
                    break

        return result
```

**Why `i == cur_distance` triggers a jump:**  
Every index up to `cur_distance` belongs to "jump N". When `i` hits that boundary, the next step *must* cost an extra jump, so we commit, extend the window, and continue.

**Why `i < len(nums) - 1` is not needed in the loop condition:**  
The early-exit `break` fires as soon as the reachable window covers the last index, so you never process indices past it unnecessarily.

## Common Pitfalls

| Mistake | Why it fails | Fix |
|---------|-------------|-----|
| Jumping every time `next_distance` improves | Over-counts jumps; a jump is only needed at the window boundary | Increment `result` only when `i == cur_distance` |
| Forgetting early exit | Processes the last index and triggers an extra jump | Add `if next_distance >= len(nums) - 1: break` inside the `if` block |
| Using `range(len(nums) - 1)` | Misses the update for the last index in edge cases | Use `range(len(nums))` and let early exit handle termination |
| BFS with an explicit queue | Correct but O(n) space; the greedy approach needs O(1) | Use the window-based greedy instead |

---

## Key Takeaways

1. **Greedy by window, not by position.** You don't need to know *where* the optimal jump lands тАФ just how far the current window reaches.
2. **Jump count = number of window expansions.** Each `i == cur_distance` event corresponds to exactly one forced jump.
3. **Early exit is safe and important.** Once `next_distance >= n - 1`, the answer is already in `result`; no further scanning needed.
4. **Compare with LC 55 (Jump Game I).** That problem only asks *can* you reach the end (boolean); LC 45 asks *how many* jumps (greedy count). Both use the same reachability scan, but LC 45 layers the window-jump counter on top.
