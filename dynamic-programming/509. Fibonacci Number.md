# LC 509 · Fibonacci Number

## Problem Statement

The Fibonacci sequence is defined as:

```
F(0) = 0
F(1) = 1
F(n) = F(n-1) + F(n-2),  for n > 1
```

Given an integer `n`, return `F(n)`.

**Constraints:** `0 <= n <= 30`

---

## Solution — Space-Optimized DP

Instead of storing the entire DP array, we only keep track of the **last two values** using a two-element list `dp = [prev2, prev1]`, rolling it forward each iteration.

```python
class Solution:
    def fib(self, n: int) -> int:
        if n <= 1:
            return n
        
        total = 0
        dp = [0, 1]           # dp[0] = F(i-2), dp[1] = F(i-1)

        for i in range(2, n + 1):   # iterate from F(2) up to F(n), inclusive
            total = dp[0] + dp[1]   # F(i) = F(i-2) + F(i-1)
            dp[0] = dp[1]           # slide window: old F(i-1) becomes new F(i-2)
            dp[1] = total           # slide window: F(i) becomes new F(i-1)

        return total
```

> **Note:** Avoid naming the accumulator variable `sum` — it shadows Python's built-in `sum()` function.

---

## ⚠️ Common Bug — Wrong `range`

### Why `range(n)` is incorrect

`range(n)` generates indices `0, 1, 2, ..., n-1`, which is **n iterations starting from 0**.  
Inside the loop, `i` starts at `0`, but `dp` is already seeded with `F(0)` and `F(1)`.  
This means the loop **re-computes values that already exist** and **stops one step too early**.

| `range(n)` — ❌ | Actual meaning |
|---|---|
| `i = 0` | Tries to compute F(2) but thinks it's the "0th" step — off by 2 |
| Last `i = n-1` | Loop ends after computing F(n-1), **never reaches F(n)** |

For `n = 5`, `range(5)` runs 5 times, but we only need **4 new computations** (F(2) through F(5)).  
The result returned would be `F(4) = 3` instead of the correct `F(5) = 5`. ❌

---

### Why `range(2, n + 1)` is correct

We seed `dp` with the known base cases `F(0) = 0` and `F(1) = 1`.  
The loop only needs to compute the **new** values starting from `F(2)`.

| Iteration | `i` | Computes |
|---|---|---|
| 1st | 2 | F(2) = F(0) + F(1) = 1 |
| 2nd | 3 | F(3) = F(1) + F(2) = 2 |
| 3rd | 4 | F(4) = F(2) + F(3) = 3 |
| 4th | 5 | F(5) = F(3) + F(4) = 5 |

- **Start at `2`** — because `F(0)` and `F(1)` are already handled by the base case.  
- **End at `n + 1`** — because Python's `range` is **exclusive** on the right, so `range(2, n+1)` includes `n` as the last value, ensuring we compute all the way up to `F(n)`.

## Trace Table — `n = 5`

| `i` | `dp[0]` (F(i-2)) | `dp[1]` (F(i-1)) | `total` (F(i)) |
|---|---|---|---|
| — | 0 | 1 | — *(initial state)* |
| 2 | 0 | 1 | **1** |
| 3 | 1 | 1 | **2** |
| 4 | 1 | 2 | **3** |
| 5 | 2 | 3 | **5** ✅ |

---

## Complexity Analysis

| | Complexity |
|---|---|
| **Time** | O(n) — single pass from 2 to n |
| **Space** | O(1) — only two variables stored, no full array |

---

## Key Takeaways

1. **`range(2, n+1)` not `range(n)`** — always start the loop at `2` since base cases F(0) and F(1) are pre-seeded. The `+1` ensures F(n) itself is included.
2. **Rolling window DP** — storing only the last two values reduces space from O(n) to O(1) compared to a full DP array.
3. **Don't shadow builtins** — avoid variable names like `sum`, `list`, `max`, `min` in Python.
4. **Edge case first** — `if n <= 1: return n` handles both `n=0` and `n=1` before the loop starts.
