# LC 343 · Integer Break

## Problem Statement

Given an integer `n`, break it into the sum of `k` positive integers where `k >= 2`,  
and **maximize the product** of those integers.  
Return the maximum product.

**Constraints:** `2 <= n <= 58`

---

## Key Insight — Two Choices at Every Split

For any way to split `i = j + (i-j)`, you have exactly **two options**:

1. **Stop here** — use `j` and `(i-j)` as the final two pieces → product is `j * (i-j)`
2. **Break further** — use `j` as-is, but optimally break `(i-j)` into more pieces → product is `j * dp[i-j]`

You don't know in advance which is better, so you take the `max` of both.  
Iterating `j` from `1` to `i//2` tries every possible first split point.

---

## DP Definition

```
dp[i] = maximum product obtainable by breaking integer i
```

**Base cases:**
- `dp[0] = 0`, `dp[1] = 0` — can't meaningfully break 0 or 1
- `dp[2] = 1` — only split is `1 + 1`, product = `1 * 1 = 1`

**Recurrence:**
```
dp[i] = max(dp[i], j * (i-j), j * dp[i-j])    for j in range(1, i//2 + 1)
```

---

## Solution

```python
class Solution:
    def integerBreak(self, n: int) -> int:
        dp = [0] * (n + 1)
        dp[2] = 1                           # base case: break 2 → 1*1 = 1

        for i in range(3, n + 1):
            for j in range(1, i // 2 + 1):
                dp[i] = max(
                    dp[i],          # ← keep best result from previous j iterations
                    j * (i - j),    # ← split into exactly 2 pieces: j and (i-j)
                    j * dp[i - j]   # ← split into j + optimally broken (i-j)
                )

        return dp[n]
```

---

## Why Is `dp[i]` Inside the `max(...)`?

This is the most important subtlety in this solution.

The inner loop tries every split point `j = 1, 2, ..., i//2`.  
Each value of `j` produces a candidate answer, and we want the **best one across all j values**.

Without `dp[i]` in the max, each iteration of `j` **overwrites** the result from the previous one:

```python
# ❌ Without dp[i] — later j values blindly overwrite earlier (possibly better) results
dp[i] = max(j * (i-j), j * dp[i-j])   # j=1 finds something good...
dp[i] = max(j * (i-j), j * dp[i-j])   # j=2 overwrites it entirely ← BUG
```

With `dp[i]` in the max, each iteration **carries forward** the best result seen so far:

```python
# ✅ With dp[i] — acts as a running maximum across all j values
dp[i] = max(dp[i], j * (i-j), j * dp[i-j])   # j=1: compare against 0 (initial)
dp[i] = max(dp[i], j * (i-j), j * dp[i-j])   # j=2: compare against best from j=1
dp[i] = max(dp[i], j * (i-j), j * dp[i-j])   # j=3: compare against best from j=1,2
#            ↑
#     this is why dp[i] is here — it's a running best, same as: best = max(best, candidate)
```

Think of it as the same pattern you use in any search loop:
```python
best = 0
for candidate in candidates:
    best = max(best, candidate)   # ← dp[i] plays the role of "best"
```

---

## What Each Term in `max(...)` Means

```python
dp[i] = max(dp[i],  (i-j) * j,  j * dp[i-j])
#            ↑           ↑              ↑
#       running      exactly 2      j + further
#       best so      pieces,        break (i-j)
#       far          stop here      optimally
```

| Term | Meaning | When it wins |
|---|---|---|
| `dp[i]` | Best product found from previous `j` values | Always present as running max |
| `j * (i-j)` | Split `i` into exactly **2 pieces**: `j` and `i-j` | When 2 pieces is optimal (e.g. `n=4`: `2*2=4`) |
| `j * dp[i-j]` | Split into `j`, then **further break** `i-j` | When more pieces is better (e.g. `n=10`: `3*3*4=36`) |

---

## Why Only Loop `j` to `i//2`?

Because splits are symmetric — splitting `6` into `2+4` is the same product as `4+2`.  
Looping only to `i//2` avoids redundant comparisons while covering all unique splits.

```
i=6: j=1 → (1,5), j=2 → (2,4), j=3 → (3,3)
     j=4 → (4,2) ← same as j=2, redundant
```

---

## Trace Table — `n = 6`

| `i` | `j` | `j*(i-j)` | `j*dp[i-j]` | `dp[i]` after this `j` |
|---|---|---|---|---|
| 3 | 1 | 1×2=**2** | 1×dp[2]=1 | 2 |
| 3 | — | — | — | **dp[3]=2** |
| 4 | 1 | 1×3=3 | 1×dp[3]=2 | 3 |
| 4 | 2 | 2×2=**4** | 2×dp[2]=2 | 4 |
| 4 | — | — | — | **dp[4]=4** |
| 5 | 1 | 1×4=4 | 1×dp[4]=4 | 4 |
| 5 | 2 | 2×3=**6** | 2×dp[3]=4 | 6 |
| 5 | — | — | — | **dp[5]=6** |
| 6 | 1 | 1×5=5 | 1×dp[5]=6 | 6 |
| 6 | 2 | 2×4=8 | 2×dp[4]=**8** | 8 |
| 6 | 3 | 3×3=**9** | 3×dp[3]=6 | **9** ✅ |

`dp[6] = 9` → split `6 = 3 + 3`, product = `3 × 3 = 9`

## Complexity Analysis

| | Complexity |
|---|---|
| **Time** | O(n²) — outer loop O(n), inner loop O(n/2) |
| **Space** | O(n) — dp array of size n+1 |

---

## Progression — How This Fits the DP Series

| Problem | DP Type | "Previous results" used |
|---|---|---|
| LC 509 / 70 / 746 | 1D, look back 1–2 steps | `dp[i-1]`, `dp[i-2]` |
| LC 62 / 63 | 2D, look at neighbors | `dp[i-1][j]`, `dp[i][j-1]` |
| **LC 343** | **1D, inner loop over all splits** | **`dp[i-j]` for all j** |

This is the first problem where the recurrence looks back at **any** prior index (not just 1 or 2 steps), requiring a nested loop to try all split points.

---

## Key Takeaways

1. **`dp[i]` inside `max(...)` is a running maximum** — it preserves the best result across all `j` iterations; without it, each `j` blindly overwrites the previous result.
2. **Two choices at every split** — `j*(i-j)` stops at 2 pieces; `j*dp[i-j]` breaks further. Always take the max of both.
3. **Loop `j` only to `i//2`** — splits are symmetric, so going further would just repeat comparisons.
4. **`dp[i-j]` is the key reuse** — this is what makes it DP rather than brute force; previously computed optimal sub-breaks are looked up in O(1).
5. **First problem with full look-back** — unlike earlier problems that only used `dp[i-1]` and `dp[i-2]`, here you look back at any `dp[i-j]`, which is why you need the inner `j` loop.
