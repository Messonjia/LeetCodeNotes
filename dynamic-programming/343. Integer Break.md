# LC 62 · Unique Paths

## Problem Statement

A robot is located at the **top-left corner** of an `m x n` grid.  
It can only move **right** or **down** at any point.  
Return the number of **unique paths** to reach the **bottom-right corner**.

**Constraints:** `1 <= m, n <= 100`

---

## Key Insight — First 2D DP Problem

This is the first problem in the staircase DP series that uses a **2D grid** instead of a 1D array.  
The recurrence is the same idea — to reach cell `(i, j)`, you must have come from:
- `(i-1, j)` — the cell directly above (moved down), or
- `(i, j-1)` — the cell directly to the left (moved right)

So:
```
dp[i][j] = dp[i-1][j] + dp[i][j-1]
```

---

## DP Definition

```
dp[i][j] = number of unique paths to reach cell (i, j)
```

**Base cases:**
- First column (`j = 0`): only one way to reach any cell — go straight down. → `dp[i][0] = 1` for all `i`
- First row (`i = 0`): only one way to reach any cell — go straight right. → `dp[0][j] = 1` for all `j`

---

## Solution

```python
class Solution:
    def uniquePaths(self, m: int, n: int) -> int:
        # 1. Initialize dp grid of size m x n
        dp = [[0] * n for _ in range(m)]

        # 2. Seed base cases: first column and first row all have exactly 1 path
        for i in range(m):
            dp[i][0] = 1        # only way is to go straight down
        for j in range(n):
            dp[0][j] = 1        # only way is to go straight right

        # 3. Fill the rest of the grid
        for i in range(1, m):
            for j in range(1, n):
                dp[i][j] = dp[i-1][j] + dp[i][j-1]

        return dp[m-1][n-1]
```

---

## Trace Table — `m = 3, n = 3`

Build the grid step by step:

**After base case initialization:**

```
1  1  1
1  0  0
1  0  0
```

**After filling inner cells:**

| | j=0 | j=1 | j=2 |
|---|---|---|---|
| **i=0** | 1 | 1 | 1 |
| **i=1** | 1 | 2 | 3 |
| **i=2** | 1 | 3 | **6** ✅ |

Calculation trace:
- `dp[1][1]` = dp[0][1] + dp[1][0] = 1+1 = **2**
- `dp[1][2]` = dp[0][2] + dp[1][1] = 1+2 = **3**
- `dp[2][1]` = dp[1][1] + dp[2][0] = 2+1 = **3**
- `dp[2][2]` = dp[1][2] + dp[2][1] = 3+3 = **6**

---

## Visual — How Paths Flow

```
(0,0)→(0,1)→(0,2)
  ↓           ↓
(1,0)→(1,1)→(1,2)
  ↓           ↓
(2,0)→(2,1)→(2,2) ← destination
```

Every cell's count = paths flowing in from the top + paths flowing in from the left.  
The border cells only receive flow from one direction — hence all 1s.

---

## Why the Inner Loop Starts at `range(1, m)` and `range(1, n)`?

- `i=0` and `j=0` are already handled by the base case initialization.  
- Starting both loops at `1` skips the first row and first column, preventing overwriting the seeded `1` values.
- This is the 2D extension of the same `range(start, n+1)` pattern from LC 509 / LC 70 / LC 746 — always skip pre-seeded base cases.

---

## Progression — 1D → 2D DP

| Problem | DP Dimensions | Recurrence | Seeds |
|---|---|---|---|
| LC 509 Fibonacci | 1D (size n) | `dp[i] = dp[i-1] + dp[i-2]` | `dp[0]=0, dp[1]=1` |
| LC 70 Climb Stairs | 1D (size n) | `dp[i] = dp[i-1] + dp[i-2]` | `dp[1]=1, dp[2]=2` |
| LC 746 Min Cost | 1D (size n+1) | `dp[i] = min(dp[i-1]+cost, dp[i-2]+cost)` | `dp[0]=0, dp[1]=0` |
| **LC 62 Unique Paths** | **2D (m × n)** | `dp[i][j] = dp[i-1][j] + dp[i][j-1]` | **first row & col = 1** |

The core idea — current cell depends on previously computed neighbors — stays the same across all four problems.

---

## Complexity Analysis

| | Complexity |
|---|---|
| **Time** | O(m × n) — every cell is computed exactly once |
| **Space** | O(m × n) — full 2D dp grid |

> **Space optimization:** Each row only depends on the row above it, so you can reduce to O(n) using a single 1D array and updating it in-place row by row.

---

## Key Takeaways

1. **2D DP follows the same logic as 1D** — "how many ways to get here" becomes a sum of neighbors instead of two prior indices.
2. **Seed the entire first row and column** — any cell on the border has exactly 1 path (no choice of direction), so initialize all of them before filling the interior.
3. **Inner loops start at 1** — `range(1, m)` and `range(1, n)` skip the pre-seeded borders, same reasoning as starting at `2` in the 1D problems.
4. **`dp[i][j]` uses `i-1` and `j-1`** — always coming from *above* (`i-1`) or *left* (`j-1`), never below or right (robot can't move those directions).
5. **Answer is at `dp[m-1][n-1]`** — the bottom-right corner, using 0-based indexing.
