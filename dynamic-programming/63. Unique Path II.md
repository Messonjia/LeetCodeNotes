# LC 63 · Unique Paths II

## Problem Statement

Same as LC 62, but now some cells contain **obstacles** (marked `1`).  
The robot cannot pass through any obstacle cell.  
Return the number of unique paths from `grid[0][0]` to `grid[m-1][n-1]`.

**Constraints:** `1 <= m, n <= 100`, `grid[i][j]` is `0` or `1`

---

## Key Differences from LC 62

| | LC 62 Unique Paths | LC 63 Unique Paths II |
|---|---|---|
| Grid values | Not given (all passable) | 0 = open, 1 = obstacle |
| Early exit | None needed | Return 0 if start or end is blocked |
| Border seeding | All 1s | 1s until first obstacle, then stop (`break`) |
| Inner cell | Always sum neighbors | Skip if obstacle (`dp[i][j]` stays 0) |

The recurrence `dp[i][j] = dp[i-1][j] + dp[i][j-1]` is identical — obstacles are handled by simply leaving `dp[i][j] = 0` and by stopping border propagation at the first blocked cell.

---

## DP Definition

```
dp[i][j] = number of unique paths to reach cell (i, j)
          = 0  if cell (i, j) is an obstacle
```

**Base cases (with obstacle awareness):**
- First column: seed `dp[i][0] = 1` until an obstacle is encountered, then `break` — all cells below are unreachable.
- First row: seed `dp[0][j] = 1` until an obstacle is encountered, then `break` — all cells to the right are unreachable.

---

## Solution

```python
class Solution:
    def uniquePathsWithObstacles(self, obstacleGrid: List[List[int]]) -> int:
        m = len(obstacleGrid)
        n = len(obstacleGrid[0])

        # Early exit: no path possible if start or end is blocked
        if obstacleGrid[0][0] == 1 or obstacleGrid[m-1][n-1] == 1:
            return 0

        dp = [[0] * n for _ in range(m)]

        # Seed first column — stop at first obstacle
        for i in range(m):
            if obstacleGrid[i][0] == 0:
                dp[i][0] = 1      # ✅ assignment =, not comparison ==
            else:
                break             # cells below this obstacle are unreachable

        # Seed first row — stop at first obstacle
        for j in range(n):
            if obstacleGrid[0][j] == 0:
                dp[0][j] = 1      # ✅ assignment =, not comparison ==
            else:
                break             # cells to the right are unreachable

        # Fill interior
        for i in range(1, m):
            for j in range(1, n):
                if obstacleGrid[i][j] == 1:
                    continue      # obstacle: dp[i][j] stays 0, no paths through here
                dp[i][j] = dp[i-1][j] + dp[i][j-1]

        return dp[m-1][n-1]
```

---

## ⚠️ Critical Bug — `==` vs `=`

The most common mistake in this problem (and a recurring Python pitfall):

```python
# ❌ WRONG — comparison, not assignment; dp stays all 0
dp[i][0] == 1
dp[0][j] == 1

# ✅ CORRECT — actually writes the value into dp
dp[i][0] = 1
dp[0][j] = 1
```

**Why Python doesn't catch this:** `==` is a valid expression that evaluates to `True` or `False` — Python silently discards the result with no error or warning. The entire `dp` grid stays `0`, so the function always returns `0` no matter what the input is.

**How to spot it:** If your solution returns `0` for all inputs (including simple obstacle-free cases), check your initialization loops for `==` vs `=`.

---

## Why `break` in the Border Loops?

```python
for i in range(m):
    if obstacleGrid[i][0] == 0:
        dp[i][0] = 1
    else:
        break          # ← why not just continue?
```

If an obstacle appears at `(k, 0)` in the first column, the robot **cannot pass through it**.  
All cells `(k+1, 0)`, `(k+2, 0)`, ... below it are **unreachable** even if they are open,  
because the only way to reach them along the border is through `(k, 0)`.

Using `continue` instead of `break` would incorrectly seed cells below the obstacle with `1`.

```
obstacleGrid col 0:   0  0  1  0  0
                      ↑  ↑  ✗  ↑  ↑
dp[i][0] with break:  1  1  0  0  0  ✅
dp[i][0] with continue: 1  1  0  1  1  ❌ (unreachable cells get seeded)
```

## Trace Table — `obstacleGrid = [[0,0,0],[0,1,0],[0,0,0]]`

Grid layout (1 = obstacle):
```
0  0  0
0  1  0
0  0  0
```

After border seeding (no obstacles in row 0 or col 0):
```
1  1  1
1  0  0
1  0  0
```

After filling interior:

| | j=0 | j=1 | j=2 |
|---|---|---|---|
| **i=0** | 1 | 1 | 1 |
| **i=1** | 1 | ✗ (obstacle, stays 0) | 1 |
| **i=2** | 1 | 1 | **2** ✅ |

Calculation:
- `dp[1][1]` = obstacle → skip, stays **0**
- `dp[1][2]` = dp[0][2] + dp[1][1] = 1+0 = **1**
- `dp[2][1]` = dp[1][1] + dp[2][0] = 0+1 = **1**
- `dp[2][2]` = dp[1][2] + dp[2][1] = 1+1 = **2**

## Complexity Analysis

| | Complexity |
|---|---|
| **Time** | O(m × n) — every cell visited once |
| **Space** | O(m × n) — full 2D dp grid |

---

## Key Takeaways

1. **`=` not `==` in initialization** — Python silently discards comparison results; always double-check assignment operators in seeding loops. If your output is always `0`, this is the first thing to check.
2. **`break` not `continue` in border loops** — an obstacle on the first row or column blocks all subsequent border cells; stop seeding immediately.
3. **Obstacle cells stay `0`** — just `continue` past them in the inner loop; their `dp` value naturally propagates as `0` to downstream cells, cutting off those paths.
4. **Early exit saves time** — if the start or end cell is an obstacle, return `0` immediately before any DP work.
5. **Recurrence is unchanged from LC 62** — obstacle handling is entirely in the initialization and the `continue` guard; the core formula `dp[i][j] = dp[i-1][j] + dp[i][j-1]` is identical.
