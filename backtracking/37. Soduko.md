# LeetCode 37 · Sudoku Solver

## Problem Statement

Given a partially filled 9×9 Sudoku board, fill in the empty cells (marked as `"."`) such that:

- Each row contains digits `1`–`9` with no repetition
- Each column contains digits `1`–`9` with no repetition
- Each of the nine 3×3 sub-boxes contains digits `1`–`9` with no repetition

Modify the board **in-place**. A valid solution is guaranteed to exist.

---

## Why Backtracking?

Sudoku is a **constraint satisfaction problem**. There is no greedy or mathematical shortcut — we must try candidates and undo bad choices. Backtracking fits perfectly:

- Try placing `"1"`–`"9"` in each empty cell
- If a placement violates a constraint → skip it
- If we get stuck → undo the last placement and try the next candidate
- If we fill all 81 cells → done

---

## Core Insight: Constraint Tracking Sets

Instead of scanning the board on every placement attempt (O(9) per check), we maintain three sets that answer "is this number already used?" in **O(1)**:

| Structure | Size | Tracks |
|---|---|---|
| `row_used[r]` | 9 sets | numbers used in row `r` |
| `col_used[c]` | 9 sets | numbers used in column `c` |
| `box_used[b]` | 9 sets | numbers used in box `b` |

### Box Index Formula

Any cell `(row, col)` maps to one of 9 boxes via:

```
box_index = (row // 3) * 3 + col // 3
```

```
box 0 | box 1 | box 2
------+-------+------
box 3 | box 4 | box 5
------+-------+------
box 6 | box 7 | box 8
```

Example: cell `(4, 7)` → `(4//3)*3 + 7//3` = `1*3 + 2` = **box 5** ✓

---

## Initialization (Pre-filled Cells)

Before backtracking, scan all pre-filled cells and register their values into the constraint sets. Without this step, the solver would ignore existing board numbers and produce invalid solutions.

```python
for row in range(9):
    for col in range(9):
        num = board[row][col]
        if num == ".":
            continue
        row_used[row].add(num)
        col_used[col].add(num)
        box_used[(row // 3) * 3 + col // 3].add(num)
```

---

## Traversal Order

Cells are visited **left-to-right, top-to-bottom**, like reading:

```python
next_row, next_col = (row, col + 1) if col < 8 else (row + 1, 0)
```

Base case: when `row == 9`, all 81 cells have been filled → return `True`.

---

## Backtracking Pattern: Choose → Explore → Un-choose

```python
for num in map(str, range(1, 10)):   # candidates "1"–"9"
    if (num not in row_used[row]
        and num not in col_used[col]
        and num not in box_used[(row // 3) * 3 + col // 3]):

        # CHOOSE
        board[row][col] = num
        row_used[row].add(num)
        col_used[col].add(num)
        box_used[(row // 3) * 3 + col // 3].add(num)

        # EXPLORE
        if self.backtracking(next_row, next_col, ...):
            return True

        # UN-CHOOSE
        board[row][col] = "."
        row_used[row].remove(num)
        col_used[col].remove(num)
        box_used[(row // 3) * 3 + col // 3].remove(num)

return False  # no valid candidate → trigger backtrack
```

The un-choose step is critical — when backtracking, the constraint sets must be restored so sibling branches see a clean state.

## Trace Example (partial)

Suppose cell `(0, 0)` is empty. Try `"1"`:

| Step | Action | board[0][0] | row_used[0] |
|---|---|---|---|
| Choose | place `"1"` | `"1"` | `{..., "1"}` |
| Explore | recurse to `(0, 1)` | — | — |
| Dead end | no valid num found downstream | — | — |
| Un-choose | remove `"1"` | `"."` | `{...}` (no `"1"`) |
| Next | try `"2"` | `"2"` | `{..., "2"}` |

## Common Bugs to Watch For

| Bug | Symptom |
|---|---|
| `next_row = row + 1` when `col < 8` | Skips entire rows; board never fully filled |
| `board[row][col] == num` (comparison, not assignment) | Board never updated; infinite recursion |
| `num not in row_used` (missing index) | Checks if `num` is in the list object, always `False`; places every number everywhere |

---

## Complexity

| | Value | Reason |
|---|---|---|
| Time | O(9^m) worst case | `m` = number of empty cells; 9 candidates each |
| Space | O(81) = O(1) | Constraint sets hold at most 9 elements each; recursion depth ≤ 81 |

In practice, constraint pruning eliminates the vast majority of branches, making it very fast on valid Sudoku inputs.

## Complete Solution

```python
class Solution:
    def solveSudoku(self, board: List[List[str]]) -> None:
        row_used = [set() for _ in range(9)]
        col_used = [set() for _ in range(9)]
        box_used = [set() for _ in range(9)]

        for row in range(9):
            for col in range(9):
                num = board[row][col]
                if num == ".":
                    continue
                row_used[row].add(num)
                col_used[col].add(num)
                box_used[(row // 3) * 3 + col // 3].add(num)

        self.backtracking(0, 0, board, row_used, col_used, box_used)

    def backtracking(self, row, col, board, row_used, col_used, box_used) -> bool:
        if row == 9:
            return True

        next_row, next_col = (row, col + 1) if col < 8 else (row + 1, 0)

        if board[row][col] != ".":
            return self.backtracking(next_row, next_col, board, row_used, col_used, box_used)

        box_idx = (row // 3) * 3 + col // 3
        for num in map(str, range(1, 10)):
            if (num not in row_used[row]
                    and num not in col_used[col]
                    and num not in box_used[box_idx]):
                board[row][col] = num
                row_used[row].add(num)
                col_used[col].add(num)
                box_used[box_idx].add(num)

                if self.backtracking(next_row, next_col, board, row_used, col_used, box_used):
                    return True

                board[row][col] = "."
                row_used[row].remove(num)
                col_used[col].remove(num)
                box_used[box_idx].remove(num)

        return False
```

## FAQ

### Q: Why do we need `row_used`, `col_used`, and `box_used`? Why initialize them?

These three structures are a live "what's already taken" registry for each row, column, and box. Without them, every placement attempt would require scanning the board (O(9) per check). With them, each check is **O(1)**.

They must be initialized before backtracking begins because the board comes in **pre-filled** — some cells already have numbers. Those existing numbers are already constraints. If row 0 has a `"5"` pre-filled, you can't place another `"5"` anywhere in row 0, even in the very first backtracking step. Skipping initialization means the solver ignores existing board values and produces invalid solutions.

```python
row_used = [set() for _ in range(9)]  # a list of 9 empty sets
```

After initialization from a partial board they might look like:

```python
row_used = [
    {"5", "3", "7"},   # row 0 already has 5, 3, 7
    {"6"},             # row 1 already has 6
    {"9", "8", "6"},   # row 2 already has 9, 8, 6
    ...
]
```

---

### Q: What does `for num in map(str, range(1, 10))` do?

It's two familiar things combined:

- `range(1, 10)` produces integers `1, 2, 3, ..., 9`
- `map(str, range(1, 10))` applies `str()` to each, producing `"1", "2", ..., "9"`

This is necessary because the board stores **string digits**, not integers. If you iterated over `range(1, 10)` directly, the membership check `num not in row_used[row]` would compare an `int` against a set of `str` — it would never match, and the solver would place every number everywhere.

All three of these are equivalent:

```python
for num in map(str, range(1, 10)):          # compact, functional style
for num in [str(i) for i in range(1, 10)]:  # list comprehension
for num in ["1","2","3","4","5","6","7","8","9"]:  # hardcoded
```

---

### Q: Why use `set()` instead of `list` for the constraint structures?

A list would work correctly since each digit appears at most once. But `set` is the better choice for two reasons:

**1. O(1) membership check** — the core operation we need:

```python
"5" in {"5", "3", "7"}   # O(1) — set
"5" in ["5", "3", "7"]   # O(n) — list scans one by one
```

**2. Semantics** — a set models "a collection of unique items with no order", which is exactly what we're representing: *which numbers are already taken in this row/col/box*. A list implies order and allows duplicates, neither of which applies here.

| | `set` | `list` |
|---|---|---|
| Membership check | O(1) ✅ | O(n) |
| Allows duplicates | No ✅ | Yes |
| Preserves order | No (irrelevant here) | Yes (irrelevant here) |
| Supports `add`/`remove` | ✅ | ✅ |

---

## Key Takeaways

- Backtracking = **systematic trial and error with pruning** — try, go deep, undo if stuck
- Constraint sets trade **space for speed**: O(1) validity checks instead of O(9) board scans
- Always initialize constraint sets from **pre-filled cells** before backtracking begins
- The **un-choose step must mirror the choose step exactly** — every add needs a matching remove
- `map(str, range(1, 10))` produces string candidates `"1"`–`"9"` to match the board's string type
