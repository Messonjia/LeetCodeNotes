# LeetCode 491 · Non-Decreasing Subsequences

## Problem Statement

Given an integer array `nums`, return **all different possible non-decreasing subsequences** of the given array with **at least two elements**. You may return the answer in any order.

**Example:**
```
Input:  nums = [4, 6, 7, 7]
Output: [[4,6],[4,6,7],[4,6,7,7],[4,7],[4,7,7],[6,7],[6,7,7],[7,7]]
```

---

## Core Idea — Backtracking on a Decision Tree

At each level of recursion, we decide which element to append to the current `path`. The key constraints are:

1. **Non-decreasing order** — we can only append `nums[i]` if it is ≥ the last element in `path`.
2. **No duplicate branches** — at each level, if we've already explored a branch starting with value `v`, we skip any later occurrence of `v` at the same level.
3. **Valid subsequence** — we only move forward in the array (via `startIndex`), never backward.

---

## Construction Strategy

| Step | Action |
|---|---|
| Recurse | At every node where `len(path) > 1`, save a copy to `result` |
| Choose | Iterate `i` from `startIndex` to end of `nums` |
| Guard | Skip if `nums[i]` breaks non-decreasing order OR was already used at this level |
| Explore | Append `nums[i]` to `path`, recurse with `startIndex = i + 1` |
| Un-choose | `path.pop()` to restore state |

---

## Solution

```python
class Solution:
    def findSubsequences(self, nums: List[int]) -> List[List[int]]:
        result = []
        path = []
        self.backtracking(nums, 0, path, result)
        return result

    def backtracking(self, nums, startIndex, path, result):
        if len(path) > 1:
            result.append(path[:])          # save a snapshot, NOT a reference

        uset = set()                        # tracks values used at THIS level only
        for i in range(startIndex, len(nums)):
            if (path and path[-1] > nums[i]) or nums[i] in uset:
                continue                   # skip: breaks order OR duplicate branch

            uset.add(nums[i])
            path.append(nums[i])
            self.backtracking(nums, i + 1, path, result)
            path.pop()
```

## Key Mechanisms Explained

### 1. `startIndex` — sliding left boundary
```
Level 0: [4, 6, 7, 7]   full array available
Level 1:    [6, 7, 7]   after picking 4
Level 2:       [7, 7]   after picking 4 → 6
```
Passing `i + 1` each time ensures elements are picked **in order** and **never reused**.

### 2. `path and path[-1] > nums[i]` — non-decreasing guard
- `path` alone evaluates as `False` when empty → **prevents IndexError on `path[-1]`**
- Only checks `path[-1] > nums[i]` when `path` is non-empty (short-circuit evaluation)
- Skips any element that would make the sequence decrease

### 3. `uset` — level-local deduplication
- Reset to an empty `set()` at **each call**, so it only tracks the current tree level
- Prevents exploring two branches that both start with the same value at the same position
- **Different from sorting + adjacent-duplicate skipping** — here the array is NOT sorted, so we use a hash set instead

### 4. `path[:]` — snapshot copy
- `result.append(path)` would store a **reference** — future `pop()` calls would corrupt it
- `path[:]` creates a **shallow copy** of the current state

## Mistakes Made

### ❌ Mistake 1 — Condition order in the pruning check

**Wrong (original attempt):**
```python
if (path and nums[i] < path[-1]) or nums[i] in uset:
```

**Correct:**
```python
if (path and path[-1] > nums[i]) or nums[i] in uset:
```

Both conditions are **logically equivalent** (`nums[i] < path[-1]` ↔ `path[-1] > nums[i]`), so this was not actually a bug — but writing `path[-1] > nums[i]` is the more natural reading: *"if the last element in path is greater than the current candidate, skip."* Keeping the subject (`path[-1]`, i.e., the constraint anchor) on the left makes the intent clearer.

## Complexity Analysis

| | Complexity |
|---|---|

| Space | O(n) — recursion depth + path storage |

---

## Key Takeaways

- **`uset` is local to each recursive call** — it deduplicates siblings at the same tree level, not across levels.
- **Do not `return` after saving to `result`** — we want to continue exploring deeper nodes in the tree.
- **`path and path[-1] > nums[i]`** is a two-part guard: emptiness check first, then value comparison.
- This problem **cannot use the sort-then-skip-adjacent pattern** (unlike subsets/combinations) because sorting would change the original sequence structure.
