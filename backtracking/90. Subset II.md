# LeetCode 90 – Subsets II

## Problem

Given an integer array `nums` that **may contain duplicates**, return all possible subsets (the power set). The solution must not contain duplicate subsets.

**Example:**
```
Input:  nums = [1, 2, 2]
Output: [[], [1], [1,2], [1,2,2], [2], [2,2]]
```

---

## How This Differs from LC 78 (Subsets)

In LC 78, all elements are unique so every path through the recursion tree produces a distinct subset. Here, duplicate values in `nums` can cause the same subset to be generated more than once **at the same recursion depth**, so we need an extra deduplication step.

---

## Approach: Backtracking + Sort + Sibling-Level Deduplication

### Core Idea

> Sort first, then skip duplicate values **within the same loop level** (i.e., among siblings in the recursion tree). Duplicates across different depths are fine — they produce legitimately different subsets.

The key insight is *where* duplicates cause problems: not across recursion depth levels, but **horizontally** — when the same value appears as multiple choices at the same position in the loop.

### Why Sort First?

Sorting groups duplicates together, making it easy to detect and skip them in a single forward pass. Without sorting, the same value could appear at non-adjacent positions and a simple "skip if seen" check would still work (via a set), but sorting is the standard approach and also enables the alternative index-based check (see below).

---

## Solution

```python
from typing import List

class Solution:
    def subsetsWithDup(self, nums: List[int]) -> List[List[int]]:
        result = []
        path = []
        nums.sort()                             # group duplicates together
        self.backtracking(nums, 0, path, result)
        return result

    def backtracking(self, nums, startIndex, path, result):
        result.append(path[:])                  # collect at every node
        uset = set()                            # tracks values used at THIS level
        for i in range(startIndex, len(nums)):
            if nums[i] in uset:                 # duplicate sibling → skip
                continue
            uset.add(nums[i])
            path.append(nums[i])
            self.backtracking(nums, i + 1, path, result)
            path.pop()
```

---

## Alternative: Index-Based Dedup (No Set Needed)

Because the array is sorted, duplicates are adjacent. Instead of a set, you can check:

```python
if i > startIndex and nums[i] == nums[i - 1]:
    continue
```

This skips `nums[i]` if it's the same value as the previous sibling choice. The condition `i > startIndex` ensures we only skip **within the current level**, not across levels.

```python
def backtracking(self, nums, startIndex, path, result):
    result.append(path[:])
    for i in range(startIndex, len(nums)):
        if i > startIndex and nums[i] == nums[i - 1]:   # sorted → adjacent duplicates
            continue
        path.append(nums[i])
        self.backtracking(nums, i + 1, path, result)
        path.pop()
```

Both approaches are correct. The set-based version is slightly more intuitive; the index-based version is more memory-efficient.

## Recursion Tree

Input: `nums = [1, 2, 2]` (after sorting)

```
[]                              ← collected
├── [1]                         ← collected
│   ├── [1, 2]                  ← collected
│   │   └── [1, 2, 2]          ← collected (leaf)
│   └── [1, 2]  ✗ SKIPPED      ← duplicate sibling (2 already used at this level)
├── [2]                         ← collected
│   └── [2, 2]                 ← collected (leaf)
└── [2]  ✗ SKIPPED             ← duplicate sibling (2 already used at this level)
```

**Output:** `[[], [1], [1,2], [1,2,2], [2], [2,2]]`

## Complexity

| | Complexity |
|---|---|
| **Time** | O(n · 2ⁿ) — at most 2ⁿ subsets, each copied in O(n) |
| **Space** | O(n) — recursion depth at most n; the set `uset` is O(n) per level |

---

## Key Takeaways

- **Sort first** — groups duplicates together, enabling easy detection
- **Dedup horizontally, not vertically** — `uset` (or the index check) only blocks the same value from being picked **twice at the same loop level**; the same value at a deeper level is perfectly fine (it builds a longer subset, not a duplicate one)
- **`uset` is local to each call** — a new set is created at every recursion level, so choices made at one depth never interfere with another
- **Collect at every node** — same as LC 78; subsets are gathered before the loop, not just at leaves
