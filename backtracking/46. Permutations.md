# LeetCode Study Note
## 39. Combination Sum
`Medium` · `Backtracking` · `DFS` · `Pruning`

---

## 1. Problem Statement

Given an array of distinct integers `candidates` and a target integer `target`, return all unique combinations where the chosen numbers sum to `target`. The same number may be chosen an unlimited number of times.

**Example:**
```
Input:  candidates = [2, 3, 6, 7],  target = 7
Output: [[2, 2, 3], [7]]
```

---

## 2. Key Insight & Strategy

**Why Backtracking?**

We need to explore all possible combinations — this is a classic search problem. Backtracking lets us build combinations incrementally and abandon paths (prune) the moment they exceed the target, avoiding wasted work.

**Two critical design decisions:**
- **Sort first** → enables early `break` when `candidates[i] > remaining budget`
- **Pass `i` (not `i+1`) as `startIndex`** → allows reuse of the same element
- **Use `path[:]` (a copy) when recording result** → avoids reference aliasing bugs

---

## 3. Solution Code

```python
class Solution:
    def backtracking(self, candidates, target, total, startIndex, path, result):
        if total == target:           # Base case: found valid combo
            result.append(path[:])    # Append a COPY of path
            return

        for i in range(startIndex, len(candidates)):
            if total + candidates[i] > target:
                break                 # Pruning: sorted array, no point continuing
            total += candidates[i]
            path.append(candidates[i])
            self.backtracking(candidates, target, total, i, path, result)
            #                                             ^ i, not i+1: allow repeats
            total -= candidates[i]   # Undo choice (backtrack)
            path.pop()

    def combinationSum(self, candidates, target):
        result = []
        candidates.sort()             # MUST sort for pruning to work
        self.backtracking(candidates, target, 0, 0, [], result)
        return result
```

---

## 4. Why `break` and Not `continue`?

The array is sorted. Once `candidates[i]` causes `total` to exceed `target`, every subsequent candidate is even larger. There is no way any later element can produce a valid sum — so we `break` the entire loop, not just skip one element.

| Statement | Behavior | Correct here? |
|-----------|----------|---------------|
| `break` | Stop all remaining iterations | ✅ Yes |
| `continue` | Skip this one, try next | ❌ No — wastes time, same result |

## 6. Complexity Analysis

| Metric | Value |
|--------|-------|
| Time | O(N^(T/M)) — N candidates, T target, M min value |
| Space | O(T/M) — max recursion depth |

---

## 7. Key Takeaways

- **Sort the input** before backtracking to enable pruning with `break`
- **Use `startIndex = i`** (not `i+1`) to allow the same element to repeat
- **Always append `path[:]`** (a copy) — appending `path` directly stores a reference that mutates
- **`break` vs `continue`** — use `break` when a sorted array guarantees all remaining elements are too large
- This pattern (**sort + backtrack + prune**) is the template for all combination/subset problems
