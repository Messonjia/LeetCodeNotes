## 1. Problem Statement

Given a collection of candidate numbers (`candidates`) and a target number (`target`), find all unique combinations in `candidates` where the candidate numbers sum to `target`.

**Constraints:**
- Each number in `candidates` may only be used **once**
- The solution set must **not contain duplicate combinations**
- `candidates` may contain duplicate numbers

**Example:**
```
Input:  candidates = [10, 1, 2, 7, 6, 1, 5],  target = 8
Output: [[1, 1, 6], [1, 2, 5], [1, 7], [2, 6]]
```

## 3. The Core Challenge â€” Why Duplicates Appear

After sorting, identical values create multiple "entry points" to the **same combination**.

**Example:** `candidates = [1, 1, 2]`, `target = 3`

Without deduplication, both `candidates[0]=1` and `candidates[1]=1` independently spawn the path `[1, 2]`:

```
i=0 â†’ pick candidates[0]=1, recurse startIndex=1
    i=1 â†’ pick candidates[1]=2 â†’ [1, 2] âś“  RESULT

i=1 â†’ pick candidates[1]=1, recurse startIndex=2
    i=2 â†’ pick candidates[2]=2 â†’ [1, 2] âś“  DUPLICATE!
```

**Root cause:** Two identical values at the same tree level each spawn their own identical subtree.

---

## 4. The Fix â€” One Line of Deduplication

```python
if i > startIndex and candidates[i] == candidates[i - 1]:
    continue
```

This says:
> "If I'm at the **same recursive level** (`i > startIndex`) and this value equals the one I just tried â€” skip it, because it will produce an identical combination."

### Why `i > startIndex` and NOT `i > 0`?

- `i > 0` would skip duplicates **everywhere**, including across different depths
- `i > startIndex` only skips duplicates **at the same tree level**
- Picking the same value at a **deeper level is valid** â€” e.g., result `[1, 1]` requires both 1s

```
Level 0:  startIndex=0
  i=0 â†’ pick 1  âś… (first time seeing 1 at this level)
  i=1 â†’ skip 1  â›” i > startIndex AND candidates[1]==candidates[0]
  i=2 â†’ pick 2  âś…

Level 1:  startIndex=1 (inside the i=0 branch)
  i=1 â†’ pick 1  âś… i == startIndex, so NO skip â†’ allows [1, 1] as result
  i=2 â†’ pick 2  âś…
```

## 5. Complete Solution

```python
class Solution:
    def backtracking(self, candidates, target, total, startIndex, path, result):
        if total == target:
            result.append(path[:])      # Append a COPY of path
            return

        for i in range(startIndex, len(candidates)):
            # Skip duplicates at the same tree level
            if i > startIndex and candidates[i] == candidates[i - 1]:
                continue

            # Pruning: sorted array, all further candidates are larger
            if total + candidates[i] > target:
                break

            total += candidates[i]
            path.append(candidates[i])
            self.backtracking(candidates, target, total, i + 1, path, result)
            #                                                  ^ i+1: no reuse
            total -= candidates[i]      # Undo (backtrack)
            path.pop()

    def combinationSum2(self, candidates, target):
        result = []
        candidates.sort()               # MUST sort: enables both pruning and dedup
        self.backtracking(candidates, target, 0, 0, [], result)
        return result
```

## 7. Common Mistake â€” The Wrong Solution

```python
# âťŚ WRONG â€” missing deduplication
for i in range(startIndex, len(candidates)):
    if total + candidates[i] > target:
        break
    ...
    self.backtracking(candidates, target, total, i + 1, path, result)
```

For `candidates = [1, 1, 2]`, `target = 3` this outputs `[[1,2], [1,2]]` â€” a duplicate result, because both index 0 and index 1 hold value `1` and independently produce the same combination.

**The fix:** add `if i > startIndex and candidates[i] == candidates[i-1]: continue`

## 9. Key Takeaways

- **Sort first** â€” required for both pruning (`break`) and deduplication (`continue`) to work
- **Use `i + 1`** (not `i`) when recursing â€” each element may only be used once
- **Dedup guard** `i > startIndex and candidates[i] == candidates[i-1]` eliminates duplicate combos at the same tree level
- **`i > startIndex` not `i > 0`** â€” so that picking the same value at a deeper level (e.g., result `[1,1]`) is still allowed
- **`break` for pruning, `continue` for deduplication** â€” two different `if` checks, two different purposes
