# LeetCode 47 · Permutations II

## Problem Statement

Given a collection of numbers `nums` that **might contain duplicates**, return all possible **unique** permutations in any order.

**Example:**
```
Input:  nums = [1, 1, 2]
Output: [[1,1,2],[1,2,1],[2,1,1]]
```

---

## Core Idea — Permutation + Duplicate Pruning

This is LC 46 with one added challenge: **duplicate values in `nums` would produce identical permutations** if left unchecked. The fix requires two things:

1. **Sort `nums` first** — brings duplicate values adjacent to each other, making them detectable
2. **Skip duplicate siblings** — if `nums[i] == nums[i-1]` and the previous one is not currently in use, we are about to start a duplicate branch at this tree level → skip it

---

## Construction Strategy

| Step | Action |
|---|---|
| Pre-process | `nums.sort()` to group duplicates together |
| Base case | When `len(path) == len(nums)`, save a copy to `result` |
| Choose | Iterate `i` from `0` to `len(nums) - 1` every time |
| Guard 1 | Skip if `used[i] == True` (element already in current path) |
| Guard 2 | Skip if `nums[i] == nums[i-1]` and `not used[i-1]` (duplicate sibling branch) |
| Explore | Mark `used[i] = True`, append `nums[i]`, recurse |
| Un-choose | `path.pop()`, reset `used[i] = False` |

---

## Solution

```python
class Solution:
    def permuteUnique(self, nums: List[int]) -> List[List[int]]:
        nums.sort()                             # step 1: sort to group duplicates
        result = []
        self.backtracking(nums, [], [False] * len(nums), result)
        return result

    def backtracking(self, nums, path, used, result):
        if len(path) == len(nums):
            result.append(path[:])
            return

        for i in range(len(nums)):
            # Guard 2: skip duplicate sibling branch
            if (i > 0 and nums[i] == nums[i - 1] and not used[i - 1]) or used[i]:
                continue

            used[i] = True
            path.append(nums[i])
            self.backtracking(nums, path, used, result)
            path.pop()
            used[i] = False
```

---

## The Critical Line — Duplicate Pruning Explained

```python
if (i > 0 and nums[i] == nums[i - 1] and not used[i - 1]) or used[i]:
```

This line has two independent guards joined by `or`:

### Guard 1: `used[i]`
Same as LC 46 — skip any element already in the current path.

### Guard 2: `i > 0 and nums[i] == nums[i-1] and not used[i-1]`

This detects a **duplicate sibling branch** and skips it. Break it down:

| Condition | Purpose |
|---|---|
| `i > 0` | Prevent index out of bounds when checking `i - 1` |
| `nums[i] == nums[i-1]` | Current element is a duplicate of the previous one |
| `not used[i-1]` | The previous copy is **not** in the current path — meaning we are at the **same tree level** as when we used it |

### Why `not used[i-1]` is the key

This is the subtle part. Consider `nums = [1a, 1b, 2]` (subscripts for clarity):

```
Scenario A — used[i-1] is True:
  path already contains 1a at a deeper level
  → picking 1b now builds a DIFFERENT path structure → ALLOW

Scenario B — used[i-1] is False:
  1a was already tried and backtracked at THIS level
  → picking 1b would produce an IDENTICAL branch → SKIP
```

In plain English: **if the previous duplicate was already explored and put back, don't start the same branch again with the next copy.**

---

## Step-by-Step Trace — `nums = [1, 1, 2]`

```
nums after sort: [1a, 1b, 2]   (same value, different indices)

backtracking(path=[], used=[F,F,F])
├── pick 1a (i=0) → path=[1], used=[T,F,F]
│   ├── pick 1b (i=1) → path=[1,1], used=[T,T,F]
│   │   └── pick 2 → path=[1,1,2] ✓ saved
│   └── pick 2 (i=2) → path=[1,2], used=[T,F,T]
│       └── pick 1b → path=[1,2,1] ✓ saved
│
├── pick 1b (i=1) → nums[1]==nums[0] and not used[0] → SKIP ✗
│
└── pick 2 (i=2) → path=[2], used=[F,F,T]
    ├── pick 1a (i=0) → path=[2,1], used=[T,F,T]
    │   └── pick 1b → path=[2,1,1] ✓ saved
    └── pick 1b (i=1) → nums[1]==nums[0] and not used[0] → SKIP ✗
```

**Result:** `[[1,1,2], [1,2,1], [2,1,1]]` — exactly 3 unique permutations, no duplicates.

## Complexity Analysis

| | Complexity |
|---|---|
| Time | O(n! · n) worst case — duplicate pruning reduces actual work |
| Space | O(n) — recursion depth + `used` array + `path` |

## Comparison: LC 46 vs LC 47

| | LC 46 Permutations | LC 47 Permutations II |
|---|---|---|
| **Input** | Distinct integers | May contain duplicates |
| **Sort needed?** | ❌ No | ✅ Yes — to group duplicates |
| **`used[]` needed?** | ✅ Yes | ✅ Yes |
| **Extra pruning?** | ❌ No | ✅ Yes — skip duplicate sibling branches |
| **Pruning condition** | `used[i]` only | `used[i]` OR `(nums[i]==nums[i-1] and not used[i-1])` |
| **Tree branches** | All n! branches valid | Duplicate branches pruned early |

### The one addition from LC 46 → LC 47

LC 47 is LC 46 **plus** this single extra guard:

```python
# LC 46 pruning
if used[i]:
    continue

# LC 47 pruning — one extra condition added
if used[i] or (i > 0 and nums[i] == nums[i - 1] and not used[i - 1]):
    continue
```

Everything else — the `used[]` array, always iterating from `0`, full-length base case — is identical.

---

## Key Takeaways

- **Sort first** — without sorting, duplicate values may not be adjacent, making the `nums[i] == nums[i-1]` check unreliable.
- **`not used[i-1]` distinguishes level-duplicate from depth-duplicate** — it is what separates "we're trying the same branch again" from "we're building a legitimately different path."
- The pruning operates on **tree level** (siblings), not on the path itself — `uset` from LC 491 and this condition serve the same conceptual role but in different problem contexts.
- LC 46 → LC 47 is a minimal change: one sort + one extra `if` condition. The entire backtracking skeleton stays the same.
