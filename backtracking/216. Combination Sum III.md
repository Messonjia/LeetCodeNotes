# LeetCode 216 — Combination Sum III

## Problem Statement

Find all valid combinations of `k` numbers that sum to `n` such that:
- Only numbers `1` through `9` are used.
- Each number is used **at most once**.

Return a list of all possible valid combinations (no duplicates, any order).

**Example:**
```
Input:  k = 3, n = 7
Output: [[1,2,4]]

Input:  k = 3, n = 9
Output: [[1,2,6],[1,3,5],[2,3,4]]
```

---

## Approach: Backtracking with Pruning

### Core Idea

Use DFS/backtracking over digits `1–9`. At each step, pick a digit, add it to `path` and `currentSum`, recurse, then undo both changes. When `len(path) == k`, check if `currentSum == targetSum` and record if so.

### Pruning

Two early-exit conditions cut down unnecessary branches:

1. **Sum exceeded:** `if currentSum > targetSum: return`  
   No point continuing — all remaining digits are positive, so the sum can only grow.

2. **Loop upper bound:** `range(startIdx, 9 - (k - len(path)) + 2)`  
   At any point, `k - len(path)` more digits are still needed. If you start at `i`, the longest sequence available is `[i, i+1, ..., 9]` with length `9 - i + 1`. For a valid combination to exist: `9 - i + 1 >= k - len(path)`, which gives the upper bound `i <= 9 - (k - len(path)) + 1` (exclusive: `+2`).

---

## Final Correct Solution

```python
class Solution:
    def combinationSum3(self, k: int, n: int) -> List[List[int]]:
        result = []
        self.backtracking(n, k, 0, 1, [], result)
        return result

    def backtracking(self, targetSum, k, currentSum, startIdx, path, result):
        if currentSum > targetSum:          # prune: sum already too large
            return
        if len(path) == k:
            if currentSum == targetSum:     # valid combination found
                result.append(path[:])      # snapshot — not a reference
            return
        for i in range(startIdx, 9 - (k - len(path)) + 2):  # pruned upper bound
            currentSum += i
            path.append(i)
            self.backtracking(targetSum, k, currentSum, i + 1, path, result)
            currentSum -= i                 # ✅ undo exactly what was added
            path.pop()                      # backtrack
```

## Complexity Analysis

| | Value |
|---|---|
| **Time** | O(C(9, k) · k) — at most C(9,k) combinations, each copied in O(k) |
| **Space** | O(k) recursion depth + O(C(9,k) · k) for output |

## ⚠️ My Mistakes

### Mistake 1 — Wrong rollback for `currentSum`

```python
# Inside the for loop:
currentSum += i       # added i before recursing
...
currentSum -= 1       # ❌ always subtracts 1, not what was added
currentSum -= i       # ✅ must mirror the addition exactly
```

**Rule:** every mutation before a recursive call must be **exactly mirrored** after it.

| Before recursion | After recursion (undo) |
|---|---|
| `path.append(i)` | `path.pop()` |
| `currentSum += i` | `currentSum -= i` ← must match |

### Mistake 2 — Confused about `return` only returning one result

When `result.append(path[:])` is called and then `return` is hit, it felt like only one combination would ever be collected. 

**The key insight:** `return` exits only the **current recursive call**, not the whole program. Execution resumes at the `for` loop in the **parent call**, which continues iterating and exploring other branches. `result` is a shared list passed through every call — each branch that finds a valid combination drops it in, and the `for` loop ensures all branches are visited.

```
backtracking(path=[])
│
├── i=1 → backtracking(path=[1])
│         ├── i=2 → len==k, sum≠target, return  ← back to THIS loop
│         ├── i=3 → len==k, sum==target ✅ append, return
│         └── i=4 → sum > target, return
│
├── i=2 → backtracking(path=[2])   ← for loop continues here
│         └── ...
└── ...
```

Think of `result` as a **shared basket**: every branch of the recursion tree can add to it, and `return` just means "done with this branch, go back up."

---

## Key Takeaways

- Always **mirror mutations symmetrically** — undo exactly what you did, with the same value.
- `return` in a recursive call sends control back to the **parent's `for` loop**, not to the top-level caller.
- The `for` loop is what drives **breadth** (trying all siblings); recursion drives **depth** (building one path).
- `path[:]` creates a **snapshot** — without it, all entries in `result` would point to the same mutating list.
